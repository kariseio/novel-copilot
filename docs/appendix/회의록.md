## 1. 제품 프레이밍 (합의된 기준점)

### 비전
연재 마감과 설정 일관성이라는 웹소설 작가의 가장 큰 부담을 덜어주는 AI 창작 지원 툴이다. 작가가 머릿속에만 있던 세계관·인물·사건을 구조화(온톨로지)해 한 번 정한 설정이 작품 전체에서 어긋나지 않게 지키고(RAG 기반 일관성), 막·장 단위 스토리 골격을 자동 생성해 백지 공포 없이 빠르게 초안을 확장하도록 돕는다. 목표 사용자는 '글을 대신 써주길 바라는 사람'이 아니라 '자기 작품의 작가성을 유지하면서 더 빠르고 일관되게 연재하고 싶은 창작자'이며, 이 툴은 작가를 대체하는 자동 집필기가 아니라 설정 관리자·구조 설계 비서·기억 보조 장치로서 창작의 통제권을 작가에게 남긴 채 생산성과 품질을 끌어올린다.

### 타겟 페르소나
- **연재 마감에 쫓기는 기성 웹소설 작가 (주 5~7회 연재, 회당 5,500자)**
  - 니즈: 매일 분량을 메우는 속도, 직전 회차들과의 톤·설정·복선 연속성 유지, 빠르게 신뢰할 만한 초안 골격을 받아 손보는 워크플로
  - 페인포인트: 마감 압박으로 설정 충돌(인물 눈 색깔·과거사·능력 수치 불일치)이 댓글로 지적받음. 수백 회차가 쌓이면 과거 회차를 다시 찾아 확인할 시간이 없음. 일반 LLM은 분량은 채워도 '내 작품의 설정과 문체'를 기억하지 못해 결국 다 고쳐 써야 함.
- **데뷔를 준비하는 신인 작가 (공모전·플랫폼 투고 목표, 첫 장편 도전)**
  - 니즈: 탄탄한 장편 구조(기승전결, 막 구성, 회차별 훅) 설계 가이드, 연독률을 높이는 회차 끊기 감각, 완결까지 끌고 갈 스토리라인 설계 역량 보완
  - 페인포인트: 초반 10화는 쓰지만 중반에 이야기가 늘어지고 어디로 가야 할지 길을 잃음. 장편 구조 설계 경험이 없어 떡밥 회수·페이싱 실패. 무엇이 부족한지 피드백을 줄 사람도, 구조를 검증할 방법도 없음.
- **설정이 복잡한 판타지·이세계물 작가 (마법체계·계급·지리·세력 관계가 방대)**
  - 니즈: 인물·세력·아이템·지명·사건 간 관계를 구조적으로 관리하고 시각화, 새 설정 추가 시 기존 설정과의 모순 자동 감지, 특정 설정을 참조하는 회차를 역추적
  - 페인포인트: 설정집을 엑셀·노션·메모장에 분산 관리하다 보니 단일 진실 공급원(single source of truth)이 없음. 마법 규칙을 바꾸면 그 규칙을 쓴 과거 장면들이 다 깨지는데 일일이 추적 불가. 신규 인물·세력을 등장시킬 때 기존 세계관과 충돌하는지 확인할 수단이 없음.

### 핵심 사용자 여정
- 1. 작품 설계 시작 - 장르·소재·톤·타깃 독자(연령/플랫폼)를 입력하고 빈 프로젝트(작품 워크스페이스)를 생성한다.
- 2. 세계관·설정 정립 - 핵심 인물, 배경, 세력, 마법/능력 체계, 주요 사건을 입력하고 이들 간 관계를 온톨로지로 구조화한다. 단일 설정 저장소(SSOT)가 만들어진다.
- 3. 스토리라인 골격 생성 - 입력된 세계관·인물·목표를 바탕으로 전체 스토리를 막(Act) 단위로, 각 막을 장/회차(Scene/Chapter) 단위로 자동 골격 생성한다. 작가는 시놉시스·갈등선·떡밥 배치를 검토하고 수정한다.
- 4. 회차 초안 확장 - 특정 회차의 골격(목표·등장인물·핵심 사건)을 선택하면, 기존 설정과 직전 회차 맥락을 RAG로 참조하여 초안 본문을 생성한다. 톤·설정이 일관되게 유지된다.
- 5. 작가 퇴고·확정 - 작가가 초안을 자신의 문체로 다듬고 대사·디테일을 확장해 회차를 확정한다. 확정된 내용은 다시 RAG 인덱스와 온톨로지에 반영되어 다음 회차의 참조 소스가 된다.
- 6. 일관성 검증 - 새 회차나 새 설정을 추가할 때 기존 온톨로지와의 모순(설정 충돌, 인물 상태 불일치, 미회수 떡밥)을 자동 감지해 경고한다.
- 7. 연재·발행 - 확정 회차를 발행(또는 외부 플랫폼용으로 내보내기)하고, 연재 일정·분량 진척을 관리한다.
- 8. 회차 누적·회고 - 연재가 쌓일수록 과거 설정·복선을 검색·역추적하고, 다음 막의 골격을 갱신하며 장기 연재를 이어간다.

### MVP 가설 (증명 대상)
MVP가 증명해야 하는 단 하나의 가설은 \"작가의 설정·이전 회차를 온톨로지+RAG로 구조화해 두면, AI가 생성하는 스토리 골격과 회차 초안의 '설정 일관성'이 일반 LLM 대비 작가가 체감할 만큼 높아져, 작가가 처음부터 다시 쓰지 않고 '다듬어 쓸' 가치를 느낀다\"는 것이다. 즉 핵심은 '글의 완성도'가 아니라 '내 세계관을 기억하고 어기지 않는 일관성'이다. 따라서 MVP에는 (1) 인물·설정을 입력하는 구조화된 세계관 에디터, (2) 그 설정과 이전 회차를 참조하는 RAG 기반 회차 골격/초안 생성, (3) 작가가 골격을 확정→다음 회차 참조 소스로 누적하는 단방향 루프, 이 세 가지만 넣는다. 반대로 화려한 자동 완성 본문 품질, 멀티 플랫폼 발행, 협업, 분석 대시보드, 본문 자동 완결 등은 가설 검증과 무관하므로 뺀다. 단일 장르(예: 현대 판타지)·단일 작가·단일 작품에 한정해 일관성 가설을 빠르게 검증한다.

### Non-Goals (MVP 의도적 제외)
- 완성된 출판 수준 본문의 완전 자동 집필(작가는 골격과 초안을 받아 직접 확장·퇴고하며, 툴은 대필기가 아니다).
- 다중 플랫폼 자동 발행·연동(카카오페이지·네이버시리즈 등 외부 플랫폼 직접 송고, 결제·정산 연동).
- 다인 협업 기능(공동 집필, 편집자 코멘트, 권한 관리, 실시간 동시 편집).
- 삽화·표지·웹툰화 등 이미지/멀티미디어 생성.
- 독자 반응·조회수·연독률 분석 대시보드 및 데이터 기반 추천.
- 다국어 번역·현지화 및 모바일 전용 앱(MVP는 단일 언어·웹 우선).
- 장르 무제한 지원(MVP는 검증을 위해 한 장르에 집중하고 점진 확장).
- 음성 입력·TTS 낭독 등 입출력 모달리티 확장.

### 차별점
- 설정의 단일 진실 공급원(SSOT) + 일관성 검증: 일반 LLM 채팅은 대화 맥락이 길어지면 설정을 잊고 충돌을 일으키지만, 이 제품은 온톨로지로 인물·세력·사건 관계를 구조화해 모순을 자동 감지·경고한다. Scrivener 같은 기존 툴은 설정을 '저장'은 하지만 관계 추론이나 충돌 감지는 못 한다.
- 장기 기억으로서의 RAG: ChatGPT는 컨텍스트 윈도/세션을 벗어난 과거 회차를 기억하지 못해 매번 작가가 설정을 다시 붙여넣어야 한다. 이 제품은 확정된 모든 회차·설정·대사를 인덱싱해 수백 회차가 쌓여도 과거 맥락을 자동 참조한다.
- 구조 우선 설계(top-down 골격 생성): 일반 LLM은 프롬프트당 단발성 본문을 뱉지만, 이 제품은 막·장 단위로 전체 스토리 골격을 먼저 세우고 회차를 그 안에서 채우게 해 장편 페이싱·떡밥 배치·완결성을 관리한다. 신인 작가의 '중반 붕괴' 문제를 구조적으로 방지한다.
- 작가 통제권 유지의 워크플로: 대필 자동화가 아니라 '골격 제안 → 작가 확정 → 확정분이 다시 다음 생성의 근거로 누적'되는 인간 주도 루프를 설계해, 작가의 문체·작가성을 보존하면서 생산성을 높인다.
- 웹소설 도메인 특화: 회차(분량·훅·끊기) 단위 작업, 연재 워크플로, 떡밥/복선 추적 등 범용 글쓰기 툴이 다루지 않는 웹소설 고유의 작업 모델을 1급 개념으로 제공한다.

## 2. 도메인별 합의 결정 로그

### 2.1 온톨로지 기반 세계관 정립

**합의된 접근:** MVP는 'PostgreSQL 단일 소스 위의 그래프형 CRUD + 결정론 규칙 엔진 + 결정론 직렬화 API'로 환원되는 LLM-프리 신뢰 영역에 집중한다. 핵심은 세 가지: (1) 타입별 정형 필드를 5개 이내로 극단 제한한 구조화 엔티티/관계 에디터로 입력 마찰을 최소화하고 정형 필드 채움률을 1급 선행 지표로 본다, (2) 시점 인지형 결정론 충돌 감지(EntityStateTimeline/valid_from·to를 데이터·판정 레벨 P0로 승격, UI는 현재 상태만 입력) 4종을 evidence 강제·dismiss 학습과 함께 운영해 오탐 신뢰 붕괴를 방지한다, (3) 온톨로지→생성 도메인 연결은 정형 필드=결정론 lookup 주입 / 서사 배경=RAG 검색 주입으로 책임을 명확히 분리한 StructuredContextSnapshot 계약(동기 경로 LLM 배제, entity_revision_refs로 주입 설정 버전 재현)으로 묶는다. 충돌 감지는 엔티티 정합성=저장 시 동기, 본문 정합성=회차 확정 시 비동기 배치로 책임을 나눈다. EntityRevision은 풀 UI 없이 데이터만 처음부터 append-only로 쌓아 시점 충돌 감지와 일관성 디버깅의 전제를 확보한다. 무엇보다 온톨로지 주입 유/무 블라인드 A/B + '다시 쓴 비율' 태깅을 MVP에 내장해 가설이 회색지대에 빠지지 않게 한다. LLM 의미 충돌 감지·그래프 시각화·자동 추출 전면판·스키마 커스터마이즈·그래프 DB는 가설 입증 후로 명확히 연기해 작가 통제권 유지(인간 주도 루프)와 SSOT 단일성을 지킨다.

**핵심 결정**

| 쟁점 | 결정 | 근거 |
|---|---|---|
| 시점(temporal) 모델의 P0 승격 — 가장 큰 쟁점 | 관계와 상태에 대한 시점성(valid_from/to, EntityStateTimeline)을 P0 데이터 레벨로 끌어올린다. 단 MVP UI는 '현재 상태' 단일 입력만 요구하고, 충돌 판정은 항상 as_of_chapter 시점 기준으로 수행한다. 시점 입력을 강제하지 않되 데이터 스키마와 판정 로직은 처음부터 시점 인지형으로 만든다. | 개발 반론의 자기모순 지적('결정론 충돌 감지=P0'인데 '시점 추적=P2')을 전면 수용. 시점 없는 충돌 감지는 회상·과거 회차·변신/각성 서사에서 오탐을 쏟아내고, 작가가 한두 번 헛경고를 보면 인박스 전체를 ignore해 기능이 죽는다. 이는 successMetrics의 '충돌 감지 유효성(수용률 대 무시율)'을 직접 무너뜨려 MVP 가설 검증 자체를 불가능하게 한다. 다만 입력 마찰 우려를 반영해 UI는 단순화하고 데이터/판정 계층에서만 시점을 책임진다. |
| SSOT 삼중 진실(정형 필드 vs 자유 서술 vs 회차 본문) 정합성 | 타입별 핵심 정형 필드를 5개 이내로 극단 제한하고, '정형 필드 채움률'을 1급 선행 지표로 격상한다. 정형 필드는 결정론적 주입의 유일한 근거, 자유 서술은 RAG 검색 근거로 역할을 계약 문서에 명문화한다. 회차 본문에서 정형 필드 후보를 추출해 '한 클릭 승격' 제안하는 기능은 온보딩/콜드스타트 한정으로 P2 일부를 앞당겨 검토한다(MVP 필수는 아님). | 작가는 본능적으로 정형 필드를 비우고 묘사를 서술에 쏟는다는 지적이 타당하다. 정형 데이터가 비면 결정론 주입의 토대가 사라져 일관성 가설의 근거가 무너진다. 입력 부담을 줄이는 것이 성패이므로 필드를 최소화하고, 채움률이 낮으면 가설 검증이 불가하므로 조기 경보 지표로 삼는다. |
| 온톨로지→RAG 결합 효과의 측정 가능성 | 온톨로지 주입 유/무 블라인드 A/B 평가 프로토콜과 '설정 충돌로 다시 쓴 비율' 태깅을 MVP에 처음부터 내장한다. 동시에 결정론 충돌 감지가 잡은 건수 같은 객관 카운트 보조 지표를 병행한다. | 생성 품질의 분산이 커서 정성 인터뷰만으로는 가설이 입증도 반증도 안 되는 회색지대에 빠진다는 지적을 수용. MVP의 단 하나의 가설(일관성 체감)을 검증하려는 제품에서 측정 프로토콜 부재는 3개월 후 '잘 모르겠다'를 결론으로 만든다. 측정 설계는 기능이 아니라 MVP의 전제다. |
| StructuredContextSnapshot — 결정론 lookup vs RAG 검색 분리 | 정형 필드=결정론적 lookup 주입(프롬프트에 박아넣음), 서사적 배경=RAG 벡터 검색 주입으로 책임을 인터페이스 계약(OpenAPI/계약 문서)에 명문화한다. 동기 경로에는 LLM을 절대 넣지 않고, 스냅샷은 템플릿 문자열 기반 결정론적 직렬화로만 산출한다. | 설정값(눈 색깔=청색)은 '찾는' 대상이 아니라 '박는' 대상이라는 지적이 핵심 설계 원칙으로 정확하다. 둘을 섞으면 임베딩 검색이 핵심 설정을 누락해 일관성 가설 자체가 흔들린다. 동기 경로 LLM 배제로 비용·지연 리스크(Low)도 함께 차단한다. |
| EntityRevision 변경 이력의 P0 데이터 레벨 확보 | 변경 이력 풀 UI는 P2로 유지하되, append-only EntityRevision 데이터와 StructuredContextSnapshot의 entity_revision_refs는 P0부터 쌓는다. | '생성 시점에 어떤 설정 버전이 주입됐는지' 재현은 일관성 디버깅과 시점 충돌 감지의 전제이며, 데이터는 소급 생성이 불가능하다. UI 비용은 미루되 데이터 적재는 처음부터 한다는 절충으로 프레이밍(일관성 검증)을 지키면서 공수를 통제한다. |
| 저장소 아키텍처 — 그래프 DB 도입 여부 | MVP는 PostgreSQL 단일 소스로 간다. 관계는 인접 테이블 + 재귀 CTE(2~3 hop), 정형 필드는 jsonb(핵심 필드는 생성 컬럼 승격). 그래프 DB(Neo4j)는 미도입, 필요 시 동일 인스턴스의 Apache AGE로 향후 확장 여지만 둔다. 벡터 인덱스는 'SSOT의 파생 캐시'로 규정하고 RAG 도메인 소유로 둔다. | 단일 작품 규모에서 그래프 DB는 과하고, 별도 DB는 이중 쓰기 정합성 실패로 SSOT를 둘로 쪼개 SSOT 제품의 자기부정이 된다는 지적을 수용. openQuestions가 스스로 제기한 우려와도 일치한다. |
| LLM 의미·서사 충돌 감지(P1)의 MVP 포함 여부 | MVP에서 확실히 제외(P1 유지). 향후 도입 시 라벨링 평가셋으로 precision을 먼저 측정하고, detection_method 컬럼으로 결정론/LLM 경고를 UI·신뢰도에서 분리하며 근거 문장 인용을 강제하고 자동 해결을 금지한다. | 확률적 충돌 감지의 환각·오탐이 결정론 경고의 신뢰까지 오염시킨다. MVP 가설(일관성)은 결정론 감지만으로 검증 가능하므로 검증 전 투입은 비용 대비 위험이 크다. |
| EntityReference(엔티티-회차 역참조) 채움 방식과 도메인 경계 | MVP는 작가 수동 태깅 + 회차 확정 시 name/alias 문자열 매칭 후보 제안 + 작가 승인(confirmed)으로 한정. 자동 확정은 금지. chapter_id 소유권은 생성/RAG 도메인, 엔티티 링킹 책임 경계를 RAG 팀과 문서로 못 박는다. | 자동 확정은 SSOT를 오염시키고(작가 통제권 유지 프레이밍 위배), RAG 청킹과 책임 경계가 모호해지면 두 팀이 같은 데이터를 이중 관리하게 된다는 지적을 수용. |
| 인터랙티브 관계 그래프 시각화(P1) | MVP에서 제외(P1 유지). 경량 리스트 뷰 + 엔티티 상세에서 1-hop 인접 관계 표시로 대체한다. 가설 입증 후로 명확히 연기. | 데모 인상은 강하나 MVP 가설(일관성)과 무관하며 검증 전 자원 분산 위험. 기획안도 '경량 리스트 뷰로 검증 후 고도화'라 적어 합의가 쉬웠다. |
| 충돌 감지 실행 타이밍(동기 vs 배치) | 책임을 분리해 둘 다 채택: 엔티티 정합성(동일 엔티티 내부 모순, 관계 모순)은 저장 트랜잭션 내 동기 검사, 회차 본문 대 설정 충돌은 회차 확정 이벤트로 비동기 워커 배치. 모든 판정은 EntityStateTimeline 참조로 시점 기준 수행. | openQuestions의 '동기 vs 배치'는 양자택일이 아니라 책임이 다른 두 경로라는 개발 분석이 명쾌하다. 동기 경로는 가벼운 결정론 쿼리만 두어 UX·성능을 지킨다. |
| 결정론 충돌 규칙 세트의 MVP 범위 | MVP 규칙은 4종으로 고정: (1) 정형 필드값 불일치, (2) 시점 기준 상태 모순(예: 사망 이후 활동), (3) 관계 모순(동일 쌍 적대+동맹 동시), (4) 중복 엔티티 의심(이름 정규화+편집거리 임계, 경고만/자동 병합 금지). 모든 경고는 evidence(충돌한 두 값·문장·회차)를 강제 저장하고 dismiss를 기억해 동일 쌍 재경고를 억제한다. | 규칙을 손꼽을 수로 제한해야 오탐 튜닝·테스트가 가능하고, 외부 룰엔진(Drools)은 불필요. evidence 강제와 dismiss 학습은 시점 모델과 함께 오탐 신뢰 위기를 완화하는 핵심 장치다. |
| 콜드 스타트(초기 입력 부담) 대응 | 기존 설정집(노션/엑셀/텍스트) 임포트 마법사와 첫 1~2회차 본문 기반 엔티티 후보 자동 추출(검토·승인형)을 온보딩 한정으로 제한적 포함을 검토한다. 장르별 시드 스키마/예시 템플릿은 MVP에 포함. | 온톨로지·확정 회차가 비면 RAG 참조 소스도 충돌 대상도 없어 초기 가치가 0이고 입력 부담으로 이탈한다. 시드 템플릿은 저비용 필수이고, 임포트/자동추출은 온보딩 한정으로 범위를 좁혀 검증 후 확대한다. |

**MVP 스코프**
- 구조화 엔티티 에디터 (Character/Faction/Item/Location/Event/Worldrule 6종), 타입별 핵심 정형 필드 5개 이내 + 자유 서술 노트, JSON Schema 기반 폼 자동 생성
- 엔티티 간 정형 관계 정의 (소속/적대/동맹/혈연/사제/소유/위치포함/선후관계), 인접 테이블 저장, valid_from/to 시점성 데이터 레벨 포함
- EntityStateTimeline (시점 기반 상태) — 데이터/판정 레벨 P0 포함, UI는 현재 상태 단일 입력으로 단순화
- 결정론적 충돌 감지 4종(필드값 불일치/시점 기준 상태 모순/관계 모순/중복 엔티티 의심) + evidence 강제 + dismiss 기억, 충돌 인박스 UI
- 충돌 감지 실행 경로 분리: 엔티티 저장 시 동기 검사 + 회차 확정 시 비동기 배치
- 생성 도메인용 StructuredContextSnapshot 익스포트 내부 API (결정론적 템플릿 직렬화, entity_revision_refs 포함, 동기 경로 LLM 배제) + RAG 도메인과 OpenAPI 계약 합의
- 정형 필드=결정론 주입 / 자유 서술=RAG 검색 주입 책임 분리를 명시한 인터페이스 계약 문서
- 엔티티 목록/검색 뷰 + 타입 필터 + 고립·미완성(빈 핵심 필드) 설정 배지 + 1-hop 인접 관계 표시(경량 리스트)
- EntityRevision append-only 변경 이력 — 데이터 적재만(풀 UI 제외)
- EntityReference 역참조 — 작가 수동 태깅 + name/alias 매칭 후보 제안 + 작가 승인(자동 확정 금지)
- 온톨로지 주입 유/무 블라인드 A/B 평가 프로토콜 + '설정 충돌로 다시 쓴 비율' 태깅 (측정 인프라, MVP 전제)
- 정형 필드 채움률을 1급 선행 지표로 추적하는 계측
- 장르별(현대 판타지 고정) 시드 스키마/예시 템플릿
- PostgreSQL 단일 소스(jsonb + 생성 컬럼 승격 + 재귀 CTE), Project/Workspace 단위 격리(단일 작가·단일 작품)

**후속/제외(deferred)**
- 인터랙티브 관계 그래프 시각화(Cytoscape.js/React Flow) — P1, 가설 입증 후
- LLM 보조 의미·서사 충돌 감지(과거사 모순/능력 한계 위반/미회수 떡밥) — P1, 평가셋 precision 선측정·detection_method 분리·근거 인용·자동해결 금지 전제
- 엔티티 자동 추출/제안(회차 본문→온톨로지) 전면판 — P2 (단, 온보딩 한정 콜드스타트 추출과 '한 클릭 정형 필드 승격'은 제한적 앞당김 검토)
- 설정 변경 이력/버전 추적 풀 UI(특정 회차 시점 설정 상태 조회 화면) — P2 (데이터 적재는 P0로 앞당김)
- 온톨로지 스키마 커스터마이즈(사용자 정의 타입/관계/필드) — P2, MVP는 단일 장르 고정 스키마
- 외부 룰엔진(Drools 등) — 규칙이 수십 개로 늘거나 작가가 직접 규칙 정의하는 단계에서 검토
- 그래프 DB(Neo4j) — 가변 깊이 경로 탐색이 실측 병목이 된 뒤로 연기, 필요 시 Apache AGE로 점진 확장
- 엔티티 카드 임베딩/벡터 인덱싱 소유권 — RAG 도메인 소유(SSOT의 파생 캐시로 규정), 본 도메인은 공급자 역할만
- 제품 프레이밍의 비목표 전체(완전 자동 집필, 다중 플랫폼 발행, 다인 협업, 이미지/멀티미디어, 분석 대시보드, 다국어·모바일 앱, 다장르, 음성/TTS)

**개발 반론(pushback) 요지**
- '결정론적 충돌 감지'를 P0로 두면서 상태의 시점 추적(temporal status)을 P2로 미룬 것은 자기모순이다. 시점 없이 '사망 인물 활동' '필드값 두 개'를 판정하면 회상·과거 회차·변신 설정에서 오탐이 쏟아지고, 작가가 한두 번 헛경고를 보면 인박스 전체를 신뢰하지 않게 된다. EntityStateTimeline(또는 최소한 관계·상태의 valid_from/to)을 P0 데이터 레벨로 끌어올리길 권고한다. 이것 없는 충돌 감지는 데모는 되어도 프로덕션에서 죽는다.
- '단일 진실 공급원(SSOT)'이라는 명명이 오해를 부른다. 실제로는 정형 필드/자유 서술 노트/회차 본문이라는 세 소스가 공존하고, 작가는 본능적으로 정형 필드를 비우고 서술에 묘사를 쏟는다. 그러면 결정론 주입에 쓸 정형 데이터가 비어 일관성 가설의 토대 자체가 무너진다. '정형 필드 채움률'을 가장 먼저 봐야 할 선행 지표로 격상하고, 입력 마찰을 5개 이내 핵심 필드로 극단적으로 줄여라.
- 온톨로지→RAG 결합 효과를 '작가 체감(정성 인터뷰)'만으로 측정하려는 것은 위험하다. 생성 품질 분산이 커서 가설이 입증도 반증도 안 되는 회색지대에 빠진다. MVP에 온톨로지 주입 유/무 블라인드 A/B 평가 프로토콜과 '설정 충돌로 다시 쓴 비율' 태깅을 처음부터 내장해야 한다. 이게 없으면 3개월 뒤 '잘 모르겠다'가 결론이 된다.
- StructuredContextSnapshot(생성 도메인 인터페이스)을 P0 feature로 둔 것은 정확하나, '검색(RAG)'과 '정확한 lookup'을 섞으면 안 된다. 설정값(눈 색깔=청색)은 벡터 검색으로 '찾는' 대상이 아니라 프롬프트에 결정론적으로 '박는' 대상이다. 정형 필드=결정론 주입, 서사 배경=RAG 검색 주입으로 책임을 분리하지 않으면 임베딩 검색이 핵심 설정을 누락해 가설이 흔들린다. 이 분리를 인터페이스 계약 문서에 명문화하라.
- P1로 분류된 'LLM 보조 의미·서사 충돌 감지'는 MVP에서 확실히 빼는 게 맞다. 다만 이미 '오탐 관리와 신뢰도가 관건'이라고 적어놓고도 후속 일정에 막연히 둔 점이 위험하다. 도입 전 반드시 라벨링된 평가셋으로 precision을 먼저 측정하고, 결정론 경고와 UI·신뢰도 표기를 분리(detection_method)하지 않으면 LLM 오탐이 결정론 경고의 신뢰까지 오염시킨다.
- 그래프 DB(Neo4j 등) 도입을 openQuestions에서 고민하는데, 단일 작품 규모에서는 명백히 과하고 오히려 SSOT를 깨는 이중 쓰기 정합성 문제를 새로 만든다. Postgres 단일 소스 + 재귀 CTE로 충분하며, 벡터 인덱스조차 'SSOT의 파생 캐시'로 규정해야 한다. SSOT를 표방하는 제품이 진실 소스를 둘로 쪼개는 것은 자기부정이다.
- 버전/이력 추적을 통째로 P2로 미룬 것은 재고가 필요하다. 풀 UI는 P2가 맞지만, '생성 시점에 어떤 설정 버전이 주입됐는지' 재현하는 EntityRevision 데이터는 일관성 디버깅과 시점 충돌 감지의 전제라 P0 데이터 레벨에서 append-only로 남겨야 한다. 나중에 소급 생성은 불가능하다 — 데이터는 처음부터 쌓아야 한다.
- EntityReference(엔티티-회차 역참조) 채움을 자동 추출로 기대하는 흐름이 보이는데, MVP에서는 작가 수동 태깅 + 이름 매칭 후보 제안 + 작가 승인으로 한정하라. 자동 확정은 SSOT를 오염시키고, RAG 도메인의 청킹/엔티티 링킹과 책임 경계가 모호해진다. 이 책임 경계(누가 chapter_id를 소유하고 누가 링킹하는지)를 RAG 팀과 문서로 못 박지 않으면 두 팀이 같은 데이터를 이중 관리한다.
- P1의 인터랙티브 관계 그래프 시각화는 데모에서 인상적이지만 MVP 가설(일관성)과 무관하다. 경량 리스트 뷰 + 1-hop 인접 표시로 충분하다고 이미 적어놓고도 P1에 둔 것은 검증 전 자원 분산 위험. 가설이 입증된 뒤로 명확히 연기하라.

**남은 리스크**
- 정형 필드 채움률 저조 리스크: 시점 모델·필드 최소화에도 작가가 여전히 정형 필드를 비우고 묘사를 자유 서술에 쏟으면 결정론 주입 근거가 비어 가설 토대가 약해진다. 채움률 조기 경보에 의존하되 임계 미달 시 가설 검증 자체가 불가하다는 구조적 위험은 남는다.
- 시점 입력 마찰과 정확도의 트레이드오프: MVP UI를 '현재 상태'만 받기로 단순화했으므로, 회상·시간 역행 서사에서 작가가 시점 컨텍스트를 명시하지 않으면 여전히 일부 오탐이 발생할 수 있다. evidence·dismiss 학습으로 완화하나 완전 제거는 아니다.
- A/B 평가의 표본/노이즈 한계: 단일 작가·단일 작품 MVP에서는 표본이 작아 블라인드 A/B와 다시 쓴 비율 태깅에도 통계적 결론의 신뢰구간이 넓을 수 있다. 객관 카운트(결정론 감지 건수) 보조에도 정성 노이즈가 남는다.
- RAG 도메인과의 인터페이스 계약 미합의 리스크: StructuredContextSnapshot 포맷/호출 시점/엔티티 링킹 책임 경계는 별도 팀과의 합의 라운드트립에 의존한다. 계약이 늦거나 어긋나면 MVP 가설의 핵심 연결고리가 지연·이중 관리로 흔들린다.
- 콜드 스타트 잔존 위험: 임포트 마법사·시드 템플릿·온보딩 한정 자동 추출에도 초기 수십 개 엔티티 선입력 부담은 남아 초기 이탈 가능성이 있다. 온보딩 자동 추출 범위를 어디까지 앞당길지의 비용 판단이 미해결.
- 중복 엔티티 의심 판정의 잔여 오탐/누락: 이름 정규화+편집거리 임계로 후보를 좁히고 자동 병합을 금지했으나 '아리엘' vs '아리엘 폰 …' 같은 동일인 판별은 기계적으로 한계가 있어 작가 수동 병합 의존이 남는다.
- 단일 장르(현대 판타지) 고정 스키마의 일반화 위험: MVP 검증 결과가 다장르로 확장될 때 6종 타입·관계·필드 스키마가 그대로 통하지 않을 수 있어, 가설 검증의 외적 타당성은 제한적이다.

### 2.2 RAG 기반 콘텐츠 재활용

**합의된 접근:** P0를 '확정 콘텐츠 청킹·임베딩·인덱싱 → 생성 시점 자동 검색·주입(자체 얇은 ContextBuilder) → 청크 단위 참조 칩 표시·핀/제외 → 확정 후 증분 재인덱싱 단방향 루프'로 엄격히 잘라낸다. 스택은 PostgreSQL 16 + pgvector(원문·벡터·메타데이터 단일 DB 통합) + Postgres FTS(nori) BM25+벡터 RRF 하이브리드 + 온프레미스 BGE-M3(벤치 후 확정) + Sonnet/GPT-4.1급 LLM(보조는 경량) + LlamaIndex + FastAPI + Celery/ARQ·Redis 워커 + Next.js 스트리밍이다. 가설은 RAG 단독이 아니라 'RAG+일관성검증' 공동 가설로 재정의하고, RAG의 KPI는 recall/참조정확도/충돌신호로 한정한다. 콜드스타트는 세계관 SSOT 즉시 인덱싱 + 기성 작품 임포트 시나리오로 우회하며, 이를 MVP 가설 검증의 공식 시나리오로 명시한다. 직전 회차는 구조화 요약·가변 N·슬롯별 토큰 cap·프롬프트 캐싱으로 비용을 통제하고, 연속성을 위해 직전 회차 우선 슬롯은 항상 예약한다. GraphRAG·말투 자동 프로파일·인용 하이라이트는 미루되, 엔티티ID 심기와 dialogue 청크 트랙 색인은 P0에 포함해 후속 비용을 줄인다. 평가 하니스(골든셋·전수 로깅·recall@k)는 P0 필수 산출물이며, 임베딩/청킹/하이브리드 가중치 A/B 벤치를 1~2주차 게이트로 둔다. 데이터 정합성은 트랜잭셔널 아웃박스+멱등 워커+sourceId 단위 전량 삭제 재삽입으로 보장한다. 도메인 간(설정·RAG·일관성검증) 책임 경계와 엔티티ID 계약은 P0 착수 전 동결한다. 일정은 P0 한정 4~6주(백엔드/AI 1~2명 + 프론트 1명).

**핵심 결정**

| 쟁점 | 결정 | 근거 |
|---|---|---|
| MVP 가설 경계 재정의 (RAG 단독 vs RAG+일관성검증) | RAG 도메인의 단독 성공지표를 '설정 일관성 향상/재작성 비율 감소'로 두지 않는다. RAG의 KPI는 '검색 recall@k, 참조 정확도(작가가 적절하다고 평가한 비율/부적절 제외율), 누락·충돌 신호 노출'로 한정한다. '설정 일관성 체감/재작성 비율 감소'는 RAG+일관성검증 두 도메인의 공동 가설로 명시하고, 블라인드 비교(RAG ON/OFF) 실험은 두 도메인을 묶은 파이프라인 단위로 측정한다. RAG 프롬프트는 검색 설정을 산문이 아니라 '준수해야 할 제약(constraint) 블록'으로 상단·캐시 고정 주입하여 준수율을 끌어올린다. | 개발 반론의 가장 핵심적인 지적('주입 ≠ 준수')을 수용. RAG는 필요조건이지 충분조건이 아니므로, RAG 단독 KPI로 일관성 가설을 귀속시키면 검증 도메인 기여와 분리 불가하고 가설이 데이터로 증명되지 않는다. 동시에 제품 프레이밍의 핵심 가치('내 세계관을 기억하고 어기지 않는 일관성')는 그대로 유지된다 — 단지 책임을 두 도메인에 정확히 배분할 뿐이다. |
| 콜드스타트 — 검증 시나리오 전제 | 가설 검증의 1차 시나리오를 '이미 수십~수백 회차 연재한 기성 작가 작품 임포트'로 공식 채택하고 기획서에 명시한다. 신규 백지 작품은 콜드스타트 보정(세계관/설정 SSOT를 회차 0부터 즉시 인덱싱 + 직전 N회차 강제포함 비중 동적 상향)으로 작동은 시키되, MVP 기간 내 핵심 가설 검증 데이터는 임포트 시나리오에서 확보한다. 따라서 P0에 '기성 회차 대량 임포트(벌크 인덱싱) 경로'를 포함한다. | 개발 반론이 지적한 '치명적 전제'를 수용. 백지 작품으로 시작하면 인덱스가 빌 동안 RAG ON/OFF 차이가 없어 MVP 기간 내 가설 검증이 불가능하다. 임포트 시나리오는 제품 타깃(연재 마감에 쫓기는 기성 작가)과도 정확히 일치하므로 프레이밍 손실 없이 검증 가능성을 확보한다. |
| 평가 하니스의 P0 필수 산출물 승격 + 임베딩 모델 게이트 | 임베딩 모델·청킹 전략·하이브리드 가중치를 '선정 완료'로 가정하지 않는다. 작가 라벨 골든셋(50~100건: '이 장면엔 이 설정/회차가 참조돼야 함')을 1차 산출물로 만들고, BGE-M3(온프레미스, dense+sparse) vs 외부 API, 하이브리드(BM25+벡터 RRF) vs 순수벡터를 A/B 벤치한다. recall@k를 1~2주차 게이트로 두고, 이를 통과하지 못하면 후속 일정을 진행하지 않는다. 모든 RetrievalQuery/Result/GenerationRun 로깅과 오프라인/온라인 평가 하니스를 1일차부터 박는다. 저작권·프라이버시상 미공개 원고 보호를 위해 온프레미스 BGE-M3를 기본값으로 두되 벤치 결과로 최종 확정한다. | openQuestion에 모델 선정이 열려 있는데 아키텍처가 확정된 듯 잡힌 모순을 해소. 검색 recall이 낮으면 전체 가설이 모래성이 되므로 평가 하니스는 가설 검증의 전제다. 한국어 웹소설 구어체·고유명사 검색 품질은 실측 전까지 미지수라는 반론을 수용해 게이트로 강제한다. |
| 직전 N회차 주입 방식 (원문 vs 구조화 요약, 고정 N vs 가변 N) | 직전 회차는 원문 그대로 항상 주입하지 않는다. '사건·상태 변화 구조화 요약'을 색인·주입하는 것을 기본으로 하고, 원문은 작가가 핀하거나 필요 슬롯에만 넣는다. N값은 고정이 아니라 회차 누적 수에 따라 가변(초반엔 직전 회차 비중 크게=검색 의존↓, 후반엔 검색 의존↑)으로 설계한다. 토큰 예산은 [세계관 코어(고정·프롬프트 캐시) : 직전 N회차 요약 : 검색된 설정 : 검색된 과거 회차] 슬롯별 상한(cap)으로 강제한다. | 회당 5,500자×N 원문 주입은 토큰예산을 금세 잠식하는 비용 함정이라는 반론 수용. 구조화 요약 + 가변 N + 슬롯 cap + 프롬프트 캐싱으로 비용·p95 지연을 통제하면서 연속성 보장은 유지한다. 연속성(직전 회차 우선 슬롯 예약) 원칙 자체는 기획 의도대로 보존한다. |
| GraphRAG (온톨로지 결합 검색)의 P0/P1 배치 | GraphRAG(관계 1~2홉 확장 검색)는 P1로 미룬다. 단, P0 청킹 단계에서 엔티티ID를 청크 메타데이터(ChunkEntityRef)에 반드시 심어 둔다. P0 엔티티 태깅은 자유형 NER이 아니라 온톨로지 보유 엔티티명 사전 매칭을 우선하고, 미스매치는 작가가 칩에서 보정한다. 엔티티ID·관계 그래프 접근 API는 P0 착수 전 설정 도메인과의 도메인 간 계약으로 먼저 동결한다(ID는 설정 도메인이 단독 발급, RAG는 참조만). | 기획안 openQuestion 판단과 개발 반론이 일치. GraphRAG는 효용은 있으나 MVP 일관성 가설 증명에 불필요하고 엔티티 추출·관계 정합성 리스크를 P0에 끌어온다. 그러나 ID를 미리 심으면 나중 그래프 결합·필터·역추적 비용이 급감하므로 'ID 심기'만 P0로 채택하는 절충이 합리적이다. |
| 캐릭터 말투 일관성 구현 방식 (자동 프로파일 추출 vs few-shot) | P1의 'CharacterVoiceProfile 자동 추출 파이프라인'은 MVP에서 제외한다. 대신 P0에서 대사를 화자 태깅과 함께 별도 dialogue 청크 트랙으로 색인하고, 특정 캐릭터 대사 생성 시 그 캐릭터의 대표 대사 청크를 few-shot 예시로 우선 주입한다. 자동 styleTraits/lexicalMarkers 추출은 후속으로 미룬다. | 말투 일관성은 자동 프로파일 추출보다 대표 대사 few-shot 주입으로 80% 효과를 더 싸게 낼 수 있다는 반론 수용. 콜드스타트(대사 누적 전 추출 불가)와 검증 난점도 회피한다. dialogue 트랙 색인은 어차피 P0 청킹에 포함되므로 추가 비용 거의 없이 사용자 스토리(말투 흔들림 방지)를 충족한다. |
| ProvenanceLink / 인용 근거 하이라이트(생성문↔참조 정밀 매핑) | ProvenanceLink와 인용 하이라이트(P2)는 MVP에서 제외하고, '문장 단위 정확 매핑'을 약속 지표로도 쓰지 않는다. 신뢰 모델은 '청크 단위 참조 칩 표시(includedInPrompt 목록, 이미 RetrievalResult로 가능)'까지만 제공한다. 정밀 문장 매핑은 향후 실험 플래그로만 검토한다. | 현 LLM으로 생성문↔근거 정밀 매핑은 신뢰성 있게 구현 난망하며, 틀린 하이라이트는 오히려 '작가 통제권·신뢰' 가치 제안을 훼손한다는 반론 수용. 청크 단위 참조 칩만으로도 '작가가 무엇을 기억해 반영했는지 신뢰·통제'라는 사용자 스토리는 충족된다. |
| 검색 실패/공백/충돌 처리 책임 경계 | RAG는 '근거가 약함/상충함'을 신호로 노출하는 데까지만 책임진다(낮은 hybridScore 표시, 충돌 후보·중복 후보 플래깅, 참조 공백 경고). '신·구 설정 충돌 판정'과 '경고 발생' 자체는 일관성 검증 도메인 책임으로 코드·팀 차원에서 분리한다. 이 경계를 P0 착수 전 도메인 간 계약으로 못 박는다. | openQuestion과 개발 반론 모두 이 지점을 회색지대로 지적. 경계를 먼저 고정하지 않으면 통합 시 책임 공백/중복이 발생한다. RAG는 '근거 청크 제공', 검증 도메인은 '충돌 판정'이라는 단일 책임 원칙으로 정리한다. |
| 확정→재인덱싱 루프의 데이터 정합성 | 회차 확정 시 Chapter.isConfirmed=true 커밋과 reindex 이벤트를 같은 트랜잭션의 아웃박스에 기록(at-least-once)하고, 멱등 워커로 소비한다. 회차 사후 수정은 contentHash·version으로 변경분만 감지하되, 재인덱싱은 sourceId 단위로 기존 청크를 전량 삭제 후 재삽입(부분 잔존 방지). IndexState.lastIndexedChapterNo로 미반영분을 추적하고 재처리 큐를 둔다. 초안(미확정)은 영구 인덱싱하지 않되, 같은 세션의 미확정 직전 초안은 인덱싱 없이 휘발성 단기 컨텍스트로만 직접 주입(영구 기억과 분리 명시). | '확정분만 기억'이라는 신뢰 모델을 지키면서 '기억에 없는 확정 회차'/'stale 청크'를 방지. 트랜잭셔널 아웃박스+멱등 워커는 검증된 패턴이며 난도가 낮다. 미확정 초안의 휘발성 단기 참조는 연속 작업 체감 저하(Low 리스크)를 신뢰 모델 훼손 없이 해소한다. |
| 기술 스택 합의 | MVP 스택을 PostgreSQL 16(SSOT) + pgvector(HNSW, 단일 DB에 원문·벡터·메타데이터 통합) + Postgres FTS(한국어 형태소 nori/mecab-ko) 기반 BM25+벡터 RRF 하이브리드 + 온프레미스 BGE-M3 임베딩(벤치 후 확정) + Claude Sonnet/GPT-4.1급 생성 LLM(보조 호출은 경량 모델) + LlamaIndex 인덱싱·리트리벌 + 얇은 자체 ContextBuilder + FastAPI + Celery/ARQ+Redis 비동기 워커 + Next.js 스트리밍 프론트로 확정한다. 재랭커(BGE-reranker-v2-m3)는 P0에서 생략하고 참조 정확도 지표가 낮으면 P1에서 도입한다. | 단일 작가·단일 작품 규모에서 별도 벡터 인프라가 불필요하고, 원문+벡터+메타데이터를 한 DB에 두면 메타데이터 필터(chapterNo<=N, sourceType, relatedCharacterIds GIN)와 벡터검색을 한 쿼리로 결합해 단순해진다. '직전 N 강제포함+토큰예산 배분+핀/제외'라는 도메인 핵심 로직은 투명성·테스트 용이성을 위해 프레임워크에 맡기지 않고 자체 ContextBuilder로 직접 구현한다. 규모 확장(다작가/멀티테넌트) 시 Qdrant 분리 이전 경로를 열어 둔다. |

**MVP 스코프**
- 확정 콘텐츠 청킹·임베딩·인덱싱 (P0): 장면 1차 분할 → 과대 장면만 토큰 기반(300~500토큰, 10~15% 오버랩) 슬라이딩 청킹. 대사는 화자 태깅과 함께 별도 dialogue 청크 트랙으로 색인. 메타데이터(chapterNo, sourceType, relatedCharacterIds 등) 부착, 엔티티 태깅은 온톨로지 엔티티명 사전 매칭 우선
- 세계관·설정 SSOT 즉시 인덱싱 (회차 0부터 설정 RAG 작동) + 기성 회차 대량 임포트(벌크 인덱싱) 경로 — 콜드스타트 우회 및 가설 검증 시나리오 지원
- 생성 시점 자동 검색·주입(컨텍스트 빌더, P0): 직전 N회차(구조화 요약, 가변 N) 강제 포함 우선 슬롯 예약 + 하이브리드 검색(벡터+BM25 RRF, characterFocusIds·chapterNo<=현재 메타데이터 사전필터) + 슬롯별 토큰예산 cap 우선순위 절단. 자체 얇은 ContextBuilder로 구현
- 참조 출처 투명 표시 + 작가 수동 조정(P0): 청크 단위 참조 칩 목록(includedInPrompt), 개별 제외/핀(강제 주입) 토글, 토큰 사용량 표시
- 확정→재인덱싱 단방향 루프(P0): 트랜잭셔널 아웃박스 + 멱등 비동기 워커, contentHash/version 기반 증분 재인덱싱(sourceId 단위 전량 삭제 후 재삽입), IndexState로 '기억에 반영됨' 상태 노출. 미확정 초안은 휘발성 단기 컨텍스트로만 주입
- 엔티티ID를 청크 메타데이터(ChunkEntityRef)에 심기 (P0): GraphRAG 자체는 미루되 ID는 미리 심어 후속 결합 비용 절감. 설정 도메인과 엔티티ID·관계 그래프 접근 API 계약 사전 동결
- 캐릭터 대사 dialogue 트랙 색인 + few-shot 주입(P0): 자동 프로파일 추출 없이 대표 대사를 few-shot 예시로 우선 주입해 말투 일관성 확보
- 검색 신호 노출(P0): 근거 약함(낮은 score)/충돌·중복 후보 플래깅까지. 충돌 판정·경고는 일관성 검증 도메인으로 위임
- 평가 하니스(P0 필수): 작가 라벨 골든셋 50~100건, RetrievalQuery/Result/GenerationRun 전수 로깅, recall@k 오프라인 평가, 검색 p95·인덱싱 지연·재작성 비율 온라인 지표
- 임베딩 모델·청킹·하이브리드 가중치 A/B 벤치 (1~2주차 게이트): BGE-M3 vs 외부 API, 하이브리드 vs 순수벡터. recall@k를 P0 종료 게이트로

**후속/제외(deferred)**
- GraphRAG: 온톨로지 관계 1~2홉 확장 검색 (P1) — P0에는 엔티티ID 심기만 수행
- CharacterVoiceProfile 자동 추출 파이프라인(styleTraits/lexicalMarkers) (P1) — MVP는 dialogue 청크 few-shot 주입으로 대체
- 설정 키워드/시맨틱 검색 뷰(역추적·떡밥 추적 전용 UI) (P1) — P0 컨텍스트 빌더에 검색 코어는 이미 있으므로 전용 뷰는 후속
- 재랭커(BGE-reranker-v2-m3 cross-encoder) (P1) — 참조 정확도 지표가 낮을 때만 도입
- ProvenanceLink / 인용 근거 하이라이트(생성문↔참조 정밀 문장 매핑) (P2, 약속 지표 제외) — 현 LLM 신뢰성 한계로 실험 플래그로만
- 검색 파라미터/재인덱싱 관리 운영 UI (P2) — 청크 크기·top-k·하이브리드 가중치는 P0에서 IndexState/설정으로 외부화하되 전용 운영 화면은 후속
- Qdrant 등 별도 벡터 인프라로 분리 이전 — 다작가/멀티테넌트 규모 확장 시점
- 본문 편집 중 인라인 검색 호출 — P0는 회차 생성 시점 트리거에 한정, 인라인 호출은 후속 판단
- 전체 작품 재색인(임베딩 모델 교체) 운영 자동화 — 경로는 1일차 설계하되 운영 UI/대량 처리 최적화는 후속

**개발 반론(pushback) 요지**
- [가설 경계 오류] '설정 일관성 향상'을 RAG 도메인 단독 성공지표로 둔 것이 가장 큰 문제. RAG는 '관련 맥락 주입'까지만 책임지고 'LLM이 어기지 않음'은 보장 못 한다. 성공지표 '재작성 비율 감소'를 RAG ON/OFF로만 귀속시키면 검증 도메인의 기여와 분리 불가. MVP 가설을 'RAG+일관성검증'을 한 묶음으로 재정의하거나, RAG의 KPI를 '검색 recall/참조 정확도'로 한정해야 한다.
- [ProvenanceLink·인용 하이라이트는 MVP·약속에서 제외] '생성된 특정 문장이 어떤 참조에서 비롯됐는지' 정확 매핑은 현 LLM으로 신뢰성 있게 구현 불가에 가깝다. 틀린 하이라이트는 오히려 '작가 통제권·신뢰' 가치 제안을 훼손한다. P2로도 약속하지 말고, 신뢰 모델은 '청크 단위 참조 칩 표시(이미 RetrievalResult로 가능)'까지만.
- [GraphRAG는 MVP 과설계, 단 ID는 미리 심어라] 온톨로지 관계 1~2홉 확장 검색은 효용은 있으나 MVP 일관성 가설 증명에 불필요하고, 엔티티 추출·관계 정합성 리스크를 P0에 끌어온다. P1로 미루되, P0 청킹 단계에서 엔티티ID를 청크 메타데이터에 반드시 심어 두는 것만 P0로(나중 결합 비용 급감). 기획안의 openQuestion 판단과 일치.
- [말투 프로파일(P1) 콜드스타트·검증 난점] 대사가 쌓이기 전 초반엔 프로파일 추출 불가(기획안도 인지). 게다가 '말투 일관성'은 자동 추출보다 '대표 대사 few-shot 주입'으로 80% 효과를 더 싸게 낸다. 별도 프로파일 추출 파이프라인은 MVP에서 빼고, 캐릭터 대사 청크를 dialogue 트랙으로 색인해 few-shot 예시로 주입하는 것으로 대체 권고.
- [콜드스타트가 가설 검증 자체를 무력화] 초반 회차엔 인덱스가 비어 RAG ON/OFF 차이가 안 난다. '단일 작가·단일 작품으로 빠르게 검증'한다면서 신규 백지 작품으로 시작하면 수십 회차 쌓일 때까지 가설 검증 불가. 검증 시나리오를 '이미 수십~수백 회차 연재한 기성 작가 작품 임포트'로 잡아야 MVP 기간 내 의미 있는 데이터가 나온다. 이건 기획서에 명시 안 된 치명적 전제다.
- [직전 회차 '원문 주입'은 토큰·비용 함정] '직전 N회차 항상 포함'을 원문 그대로 하면 회당 5,500자×N으로 토큰예산을 금세 잠식한다. 직전 회차는 원문이 아니라 '사건·상태 변화 구조화 요약'을 색인·주입하고, 원문은 핀/필요시에만. N값은 고정이 아니라 '회차 누적 수에 따라 가변'(초반엔 크게, 후반엔 검색 의존↑)으로.
- [자체 임베딩 모델 한국어 품질을 '선정 완료'로 가정하지 말 것] openQuestion에 모델 선정이 열려 있는데 일정·아키텍처는 이미 확정된 듯 잡혀 있다. 모델·청킹·하이브리드 가중치는 작가 라벨 골든셋 벤치로 결정해야 할 '실험 대상'이며, 이 벤치를 1~2주차 게이트로 두지 않으면 후속 전부가 모래성. 평가 하니스를 P0 필수 산출물로 승격해야 한다.
- [검색 실패/충돌 처리 책임 경계] openQuestion대로 '관련 참조 부족·충돌 시' 처리는 RAG와 일관성검증 도메인 사이 회색지대다. RAG는 '근거가 약함/상충함'을 신호로 노출(낮은 score, 충돌 후보 표시)까지만, '충돌 판정·경고'는 검증 도메인이. 이 경계를 코드·팀에서 먼저 못 박지 않으면 통합 시 책임 공백/중복 발생.

**남은 리스크**
- 주입 ≠ 준수: 검색·주입이 완벽해도 LLM이 제약 블록을 무시하거나 새 설정을 환각할 수 있어, RAG+일관성검증을 묶어도 준수율이 작가 체감 임계치에 못 미칠 가능성. constraint 블록·캐시 고정 주입으로 완화하나 잔존
- 한국어 웹소설 구어체·고유명사 검색 품질이 골든셋 벤치에서 recall@k 게이트를 통과 못 할 위험 — 통과 실패 시 임베딩 모델·청킹 재설계로 전체 일정 지연
- 골든셋 품질·규모 의존성: 작가 라벨 50~100건이 도메인을 충분히 대표하지 못하면 모델 선정·튜닝이 편향될 수 있음. 라벨링 작업이 작가 부담을 유발
- 기성 작품 임포트 시나리오의 데이터 확보·저작권/프라이버시 처리 — 협력 작가의 미공개 원고 대량 임포트가 현실적으로 가능한지, 온프레미스 임베딩으로 보호 가능한지 실무 확인 필요
- 엔티티 사전 매칭 태깅의 정확도 한계(동명이인·별칭·표기 변형)로 characterFocus 필터가 누락/오태깅될 수 있음 — 작가 수동 보정에 의존
- 도메인 간 계약(엔티티ID 체계, 충돌 판정 책임 경계)이 통합 단계에서 변경·미스매치될 위험 — 버전드 인터페이스로 완화하나 조직적 합의 필요
- 비용·지연 잔존: 구조화 요약·캐싱·슬롯 cap에도 불구하고 수백 회차 누적 후 검색+대용량 컨텍스트+장문 생성의 p95 지연과 토큰 비용이 작가 워크플로 체감 임계치를 넘을 가능성
- 증분 재인덱싱의 엣지 케이스: 회차 사후 대량 수정·엔티티 재정의 시 sourceId 단위 삭제 재삽입이 일시적 stale 윈도를 만들 수 있고, 아웃박스 at-least-once의 중복 처리 멱등성 검증 필요
- 가변 N·슬롯 배분의 최적값 미확정: 회차 누적 수에 따른 N·슬롯 비중은 골든셋·실측으로 튜닝해야 할 열린 파라미터로, 초기값이 부적절하면 연속성/일관성 체감 저하

### 2.3 스토리 구조 자동 생성

**합의된 접근:** 이 도메인은 자체 온톨로지·벡터DB를 소유하지 않는 '구조 오케스트레이터'로 책임을 좁힌다. 세계관 온톨로지 도메인과 RAG/회차초안 도메인의 읽기 API를 소비하고, 확정 이벤트를 outbox/event로 발행해 인덱싱·상태 갱신을 위임한다. 저장소는 PostgreSQL 16 단일(인접 테이블로 구조 데이터·온톨로지 읽기 캐시, pgvector 소량, JSONB provenance)로 통합한다.

핵심 P0 루프: (1) 구조 템플릿 + 세계관 요약을 LLM에 넣어 막 골격을 생성하되 '다음 막만 점진 생성'을 기본으로 하고, (2) 선택한 막을 회차(ChapterBeat) 비트로 분해하며(회차 수만 제어), (3) 작가가 인라인 편집·재정렬·확정한 골격만 회차 초안 생성으로 핸드오프한다. 모든 생성은 tool-calling/JSON schema + 서버측 검증 + 1회 교정 리트라이로 편집 가능한 구조화 데이터로 안정 수신하고, 세계관/시스템 프리픽스 프롬프트 캐싱과 비트 분해 모델 다운그레이드로 비용을 통제한다. 진짜 난도인 'UI 양방향 동기화 + 편집 보존 머지'를 P0 최우선 엔지니어링으로 둔다.

일관성 점검은 2단으로 분리한다: 결정론 레이어(ID 무결성·죽은 인물 재등장·떡밥 도달성)만 'High 신뢰 경고'로 P0에 넣고, LLM 의미 모순은 P1·'low-신뢰 제안'으로 톤다운해 UI에서 구분(detection_method)한다. 떡밥은 수동 태깅+상태 카운트만, 재동기화는 작가 명시 요청 시 diff 제안만 한다. 재생성은 confirmed를 절대 덮어쓰지 않고 새 proposed 후보로 병치(낙관적 락+append-only provenance)해 작가 통제권을 보존한다. 환각 엔티티는 사후 ID 매칭으로 차단하고 미매칭은 '신규 후보'로 작가 확인을 받는다.

지표는 골격 채택률·구조 수정 깊이·골격→초안 전환율·BeatToDraftLink divergence·결정론 모순 조기 발견 건수로 한정하고, '작가 체감 일관성'은 RAG/온톨로지 도메인과 공동 책임으로 표기한다. MVP 규모는 P0+톤다운된 일부 P1 기준 1인 풀스택+1인 AI 엔지니어로 8~12주(상대 도메인 API 계약 미정 시 +2~3주 버퍼).

**핵심 결정**

| 쟁점 | 결정 | 근거 |
|---|---|---|
| 골격 생성 단위 (openQuestion 1) | '다음 막만 점진 생성(act-by-act)'을 기본 정책으로 못박는다. 전체 스토리라인을 막+회차까지 한 번에 펼치는 일괄 생성은 금지(또는 명시적 고급 옵션으로만). 첫 진입 시 전체 막의 시놉시스 수준 개요(고비용 회차 비트 분해 없이)는 1회 제공해 작가가 큰 그림을 보게 하되, 회차 비트 분해는 작가가 선택한 막에 한해 수행한다. | 장편은 전개가 미정이고, 일괄 펼침은 토큰·지연·환각이 비선형으로 증가(개발 리스크 Med, pushback 7). 동시에 신인 작가의 '중반 붕괴'는 큰 그림 부재에서 오므로, 비용이 낮은 막-개요 수준의 전체 조망은 제품 가치상 유지한다. 비용은 회차 비트 분해(다발 호출)에서 발생하므로 그 부분만 점진화하면 가치와 비용을 동시에 만족한다. |
| 구조 단계 일관성 점검의 분리 (pushback 2, 리스크 High) | '구조 단계 일관성 점검'을 단일 기능에서 두 개의 별개 기능으로 재정의한다. (a) 결정론 점검 — EntityReference ID 무결성, 죽은/미등장 인물 참조, 떡밥 open 상태 도달성 — 만 P0로 편입하고 'High 신뢰 경고'로 노출. (b) LLM 의미 모순 감지(마법규칙 위반 등)는 P1로 미루고, MVP에 들어가더라도 'low-신뢰 제안(suggestion)'으로 톤다운하며 ConsistencyFlag.detection_method=llm_inferred로 UI에서 시각적으로 분리한다. | 개발 반론 수용. 의미 수준 모순은 자연어 사건을 LLM으로 명제화해 대조해야 해 재현율·정밀도가 낮고, 거짓경보(작가 피로)와 미탐(신뢰 상실)이 동시 발생. 둘을 같은 '경고'로 묶으면 신뢰도 높은 결정론 경고까지 불신을 받는다. 차별점(SSOT+일관성)은 '결정론 점검은 확실하게, 의미 점검은 겸손하게'로 재포지셔닝하면 프레이밍 손상 없이 신뢰를 지킨다. |
| 떡밥/복선 트래킹의 P0 승격 여부 (openQuestion 7, pushback 1) | 자동 떡밥 회수 추적의 P0 승격은 기각. MVP에는 '작가 수동 태깅 기반 PlotThread(seed/reinforce/resolve 역할 조인) + open/resolved 단순 상태 카운트 + 미회수 목록 표시'만 P0로 넣는다. 회수 시점 자동 감지는 P1 유지. | 개발 반론 수용. 떡밥은 자연어로 골격에 흩어져 있고 회수 판정이 의미 추론이라 자동 감지 정밀도가 낮다. 다만 '미회수 떡밥을 구조 단계에서 보이게 한다'는 핵심 가치는 수동 태깅+결정론적 도달성(seed는 있는데 resolve 회차가 없음) 카운트만으로도 상당 부분 달성되므로, 가치를 버리지 않으면서 검증 못 할 자동 추론은 뺀다. |
| 확정 회차 반영 골격 재동기화의 트리거 (openQuestion 6, pushback 3) | 자동 트리거는 MVP에서 제외. '작가 명시 요청 시에만 diff 제안'을 기본으로 한다. 재동기화 제안은 절대 confirmed 골격을 in-place로 덮어쓰지 않고 새 proposed 후보로만 생성하며, 머지는 작가 승인 게이트를 거친다. 자동 재동기화는 연재 데이터가 쌓인 뒤로 미룬다. | 확정 즉시 자동 변경은 작가가 손본 구조를 흔들어 통제권(핵심 가치)을 깨뜨린다. BeatToDraftLink.beat_snapshot을 기준선으로 한 diff 계산 인프라는 P0에 두되, 트리거는 수동·제안 전용으로 한정해 작가 주도 루프를 보존한다. |
| 분량(5,500자) 골격 단계 제어 (openQuestion 8, pushback 4) | 분량 정밀 제어를 골격 파라미터에서 제거. 골격 단계에서는 '회차 수'만 제어한다. target_length 필드는 유지하되 '강제'가 아닌 '회차 수 산정 힌트'로만 사용한다. 자수 적합성에 따른 회차 분할/병합 제안은 초안 생성(RAG 도메인) 이후 사후 피드백으로 처리한다. | 분량은 본문(초안) 단계 변수이지 골격 변수가 아니다. 골격에서 자수를 강제하면 회차 경계가 인위적으로 잘려 비트 분해 품질이 떨어진다. 개발 반론 전면 수용. |
| 도메인 책임 경계 — 온톨로지/벡터DB 소유 여부 (openQuestion 4·5, pushback 5) | 이 도메인은 자체 벡터스토어·온톨로지를 소유하지 않는다. 세계관 온톨로지 도메인과 RAG/회차초안 도메인의 '읽기 API'만 소비하고, '확정 이벤트'를 outbox/event로 발행해 인덱싱·온톨로지 상태 갱신을 위임한다. 모순 점검 중 그래프 사실 판정은 세계관 도메인에 질의한다. 이 도메인은 '구조 오케스트레이터'로 책임을 좁힌다. | 자체 그래프·검색을 들고 가면 세계관·RAG 도메인과 인프라가 중복되고 정합성이 깨진다(openQuestion 4·5의 정답). 명시적 인터페이스 계약으로 책임 중복을 제거한다. 단, 이 계약이 상대 도메인 준비도에 종속되므로 인터페이스 합의를 선행 작업으로 둔다. |
| 저장소/스택 — 그래프DB 선제 도입 여부 (pushback 6) | Neo4j 등 그래프DB 선제 도입 기각. MVP는 PostgreSQL 16 단일 저장소로 통합(인접 테이블 entity/edge로 온톨로지 읽기 캐시, pgvector로 소량 벡터, JSONB로 provenance/템플릿 메타). 다홉 관계 추론 필요성이 실측으로 입증된 후에만 그래프DB 재검토. | 단일 작품 규모에서 Act/Beat/Thread는 순서·계층·상태를 가진 정형 데이터로 parent_id/order_index 인접 리스트면 충분. 그래프DB는 운영 부담만 늘린다. 개발 스택 권고 전면 수용. |
| 골격 재생성 시 작가 편집 보존 + 분기 후보 메커니즘 (pushback 8·9, 리스크 High) | 재생성·재동기화는 절대 confirmed를 덮어쓰지 않고 항상 새 proposed 후보로 병치한다. ChapterBeat에 낙관적 락(version)과 append-only GenerationProvenance를 적용한다. 화려한 A/B 비교 UI는 P2로 미루되, 백엔드의 '새 proposed 후보 생성·diff·작가 승인 머지' 최소 메커니즘은 P0에 흡수한다. | 재생성이 작가가 손본 confirmed 골격을 유실시키면 작가성·신뢰의 핵심 가치가 파괴된다(High 리스크). '분기 후보'는 별 기능이 아니라 재생성 안전장치의 일부이므로 최소 형태를 P0에 넣어야 편집 보존 요구를 충족한다(pushback 9 수용). |
| 골격↔초안 연결의 강도 (openQuestion 3) | 확정(confirmed) 골격만 RAG/초안 도메인에 구조화 입력으로 전달하며, '구속 입력(강제)'이 아닌 '구조화된 가이드'로 전달한다. 핸드오프 시점의 beat_snapshot을 BeatToDraftLink에 동결해 이후 본문과의 divergence 계산 기준선으로 삼는다. 초안이 골격을 벗어났을 때 어느 쪽을 진실로 볼지는 작가 승인(재동기화)으로 결정한다. | 작가 주도 루프 원칙상 골격은 제안이지 강제가 아니다. 다만 추적 가능성을 위해 핸드오프 시점 골격을 동결해두면 divergence를 정량화하고 채택률 함정(리스크 Low)을 교차 검증할 수 있다. |
| MVP 성공지표 귀속 — '작가 체감 일관성' (pushback 8) | 이 도메인 고유 지표는 '골격 채택률 + 구조 수정 깊이 + 골격→초안 전환율 + BeatToDraftLink divergence + 결정론 모순 조기 발견 건수'로 한정한다. '작가 체감 일관성'은 RAG/온톨로지 도메인과의 공동 책임 지표로 표기하고, 정성 피드백은 1급 지표로 병행하되 단독 귀속하지 않는다. 채택률은 divergence·수정 깊이와 반드시 교차 측정한다. | 일관성의 실제 동력은 RAG/온톨로지 주입 품질이므로 이 도메인 단독으로 가설을 증명·반증할 수 없다(개발 (C) 정합 지적). 채택률 단독은 '대충 확정 후 본문에서 갈아엎기'를 녹색으로 보이게 하는 함정이므로 divergence 교차 지표가 필수. |
| 환각 엔티티 ID 처리 (리스크 Med) | 생성 출력의 모든 엔티티는 사후 ID 매칭을 강제한다. 온톨로지에 매칭되지 않는 인물/설정은 자동 생성 금지하고 '신규 후보'로 표시해 작가 확인을 요청한다. 미등록 사실 주장은 GenerationProvenance에 기록해 추적한다. | LLM이 그럴듯한 가짜 인물/설정을 골격에 넣어 EntityReference가 허위로 채워지는 것을 막아야 SSOT 무결성이 유지된다. 자동 생성을 막고 작가 게이트를 두는 것이 작가 통제권 원칙과도 일치. |
| 구조화 출력 안정성 (개발 (A) 진짜 난도) | LLM tool-calling / JSON schema 모드 + 서버측 Pydantic 검증 + 스키마 위반 시 1회 자동 교정 리트라이를 P0 핵심 엔지니어링으로 명시한다. 자유 텍스트 파싱은 금지. 작가가 손댄 골격을 재생성 시 보존하는 머지 로직을 동급 우선순위로 둔다. | 개발 검토의 핵심 통찰 — 진짜 난도는 '생성'이 아니라 '편집 가능한 구조화 데이터로 안정 수신 + UI 양방향 동기화 + 편집 보존 머지'에 있다. 여기가 P0 안정성을 좌우한다. |

**MVP 스코프**
- 구조 템플릿 기반 막(Act) 골격 생성 — 현대 판타지 1장르, 소수 템플릿(예: 영웅의 여정/3막/기승전결), 웹소설 회차 패턴(훅·절단·떡밥 주기) stage 가이드 주입 (P0)
- 막 → 회차(ChapterBeat) 비트 분해 — 작가가 선택한 막 단위 점진 생성, 회차 수만 제어(분량은 힌트), 회차 목표·등장인물·핵심 사건·시작/끝 상태·끊기 훅 포함 (P0)
- 골격 인라인 편집·재정렬(드래그)·추가/삭제·확정 상태 전환 — 골격 보드(트리/타임라인) + 회차 상세 패널, 낙관적 업데이트·서버 동기화 (P0)
- 확정 골격 → 회차 초안 생성 핸드오프 — confirmed beat만 구조화 입력으로 RAG/초안 도메인에 전달, BeatToDraftLink로 추적 링크 + beat_snapshot 동결 (P0)
- 세계관 온톨로지 참조 기반 골격 생성 — 온톨로지/RAG 읽기 API 소비로 인물·세력·현재 상태를 소프트 제약으로 프롬프트 주입, 생성 후 EntityReference ID 매칭 강제 (P0)
- 결정론 일관성 점검만 P0 — EntityReference ID 무결성, 죽은/미등장 인물 참조, 떡밥 open 도달성을 'High 신뢰 경고'로 노출 (ConsistencyFlag.detection_method=deterministic) (P0)
- 수동 떡밥(PlotThread) 태깅 — 작가가 seed/reinforce/resolve 역할로 태깅, open/resolved 단순 카운트 + 미회수 목록 표시 (P0, 자동 감지는 제외)
- 재생성 안전장치 — 재생성/재동기화는 confirmed를 덮어쓰지 않고 새 proposed 후보로 병치, 낙관적 락(version) + append-only provenance + 작가 승인 머지 (백엔드 최소 메커니즘 P0)
- 구조화 출력 강제 인프라 — tool-calling/JSON schema + 서버측 검증 + 1회 자동 교정 리트라이, 프롬프트 캐싱(세계관/시스템 프리픽스), GenerationProvenance 기록 (P0)
- 관측/평가 — LLM 트레이싱 + 현대 판타지 골든셋 기반 회귀 평가, 토큰/지연/비용 호출 단위 추적 (상시 병행)

**후속/제외(deferred)**
- LLM 기반 의미 수준 모순 자동 감지(마법규칙 위반 등) — P1, MVP 편입 시에도 'low-신뢰 제안'으로 톤다운·UI 분리(detection_method=llm_inferred)
- 떡밥 회수 시점 자동 감지·추적 — P1 (MVP는 수동 태깅+상태 카운트만)
- 확정 회차 반영 골격 자동 재동기화 — P1, 자동 트리거는 연재 데이터 누적 후로 (MVP는 작가 명시 요청 시 diff 제안만)
- 전체 스토리라인 막+회차 일괄 펼침 — 제외(점진 생성이 기본). 막-개요 수준 1회 조망만 허용
- 분량(5,500자) 정밀 골격 제어 — 제거, 초안 이후 사후 분할/병합 제안으로 대체
- 페이싱·긴장도 곡선 시각화 — P2
- 다중 구조 템플릿·다장르·사용자 정의 템플릿 확장 — P2 (MVP는 현대 판타지 1장르)
- 분기/대체 골격 비교의 화려한 A/B UI — P2 (백엔드 후보 병치 최소 메커니즘만 P0 흡수)
- Neo4j 등 그래프DB 도입 — 다홉 추론 필요성 실측 입증 후 재검토
- 자체 벡터스토어·온톨로지 소유 — 제외(읽기 API 소비 + 이벤트 발행으로 위임)
- 완성 본문 자동 집필, 멀티 플랫폼 발행, 협업, 분석 대시보드 — 제품 nonGoals(도메인 무관)

**개발 반론(pushback) 요지**
- 떡밥 트래킹을 MVP에서 P0로 올리자는 openQuestion에 반대한다. '신뢰할 수 있는 떡밥 회수 추적'은 떡밥이 자연어로 골격에 흩어져 있고 회수 시점 판정이 의미 추론이라 정밀도가 낮다. P0로 올리면 검증 못 할 기능에 공수를 쏟고 거짓경보로 작가 신뢰를 깎는다. MVP에서는 떡밥을 '자동 추적'이 아니라 '작가가 수동 태깅하는 PlotThread + open 상태 단순 카운트'로 한정하고, 자동 회수 감지는 P1 유지.
- '구조 단계 일관성 점검'을 단일 기능처럼 둔 것을 재정의해야 한다. 결정론 점검(ID 무결성·죽은 인물 재등장·떡밥 도달성)과 LLM 의미 모순 감지는 신뢰도·구현난도가 완전히 다른 별개 기능이다. 전자만 P0로 올리고 후자는 'low-신뢰 제안'으로 분리·명시하라. 둘을 같은 '경고'로 묶으면 작가가 결정론 경고까지 불신하게 된다.
- '확정 회차 반영 골격 재동기화(P1)'의 자동 트리거는 MVP에서 위험하다. 확정 즉시 자동으로 골격을 바꾸면 작가가 손본 구조가 흔들리고 통제권(핵심 가치)이 깨진다. MVP는 '작가 명시 요청 시에만 diff 제안', 자동 트리거는 데이터가 쌓인 뒤로 미뤄라.
- 분량 5,500자 정밀 제어를 골격 단계 파라미터로 두는 것을 빼라. 분량은 본문(초안) 단계 변수이지 골격 변수가 아니다. 골격에서 자수를 강제하면 회차 경계가 인위적으로 잘린다. MVP는 '회차 수'만 제어하고 분량은 초안 이후 사후 분할/병합 제안으로 처리.
- 이 도메인이 온톨로지/벡터DB를 직접 소유하려는 설계 유혹을 경계하라. 기획안의 '세계관 온톨로지 참조 골격 생성'과 '구조 일관성 점검'이 자체 그래프·검색을 들고 가면 세계관·RAG 도메인과 인프라가 중복되고 정합성이 깨진다. 본 도메인은 두 도메인의 읽기 API 소비 + 확정 이벤트 발행만 하도록 책임을 좁혀라(openQuestion 4·5의 정답).
- Neo4j 등 그래프DB의 선제 도입에 반대한다. 단일 작품 규모에서 Act/Beat/Thread는 PostgreSQL 인접 테이블로 충분하고, 다홉 관계 추론이 실측으로 필요해지기 전 그래프DB는 운영 부담만 만든다. MVP는 pgvector 포함 단일 PostgreSQL로 통합하라.
- '전체 스토리라인을 한 번에 막+회차까지 일괄 펼침'을 기본값으로 삼지 마라. 장편은 전개가 미정이고, 일괄 생성은 비용·지연·환각이 비선형으로 커진다. '다음 막만 점진 생성'을 기본 정책으로 못박아라(openQuestion 1의 정답).
- 이 도메인의 MVP 성공지표에 '작가 체감 일관성'을 직접 귀속시키는 것을 재검토하라. 일관성의 실제 동력은 RAG/온톨로지 주입 품질이라 이 도메인 단독으로는 가설을 증명·반증할 수 없다. 본 도메인 고유 지표는 '골격 채택률·구조 수정 깊이·골격→초안 전환율·결정론 모순 조기 발견 건수'로 한정하고, 일관성 체감은 RAG 도메인과 공동 책임으로 표기하라.
- '분기/대체 골격 비교(P2)'를 P2로 미루지만, 재동기화·재생성 시 '기존 confirmed 보존' 요구를 만족시키려면 사실상 '병치 후보' 메커니즘이 P0에서 일부 필요하다. 즉 별 기능이 아니라 재생성 안전장치의 일부로 최소 형태를 P0에 흡수해야 한다 — 별도 화려한 A/B UI는 P2로 두되, 백엔드의 '새 proposed 후보 생성·머지' 기반은 P0.

**남은 리스크**
- 상대 도메인 종속 — 온톨로지·RAG 도메인이 합의된 읽기 API/확정 이벤트 계약을 제때 제공하지 못하면 골격 생성의 일관성 핵심이 '예쁜 에디터'에 그치고 일정이 +2~3주 미끄러진다. 인터페이스 합의를 선행 마일스톤으로 둬도 외부 종속은 남는다.
- MVP 가설의 단독 증명 불가 — 이 도메인은 일관성 체감의 실제 동력(RAG/온톨로지 주입 품질) 위에 얹힌 구조 레이어라, 가설 검증이 RAG 도메인 품질에 종속된다. 공동 지표화로 책임은 분리했으나 '이 도메인 덕분에 일관성이 올랐다'를 분리 입증하기는 여전히 어렵다.
- 결정론 점검의 낮은 커버리지 — 신뢰도를 위해 의미 모순을 빼면, 정작 작가가 가장 두려워하는 '마법 규칙·설정 충돌' 같은 의미 수준 모순은 MVP에서 못 잡는다. 차별점 광고와 실제 잡히는 범위 사이의 기대 격차가 첫인상에서 실망으로 이어질 수 있다.
- 콜드스타트 — 신인 작가(타깃)일수록 온톨로지·이전 회차 입력이 빈약해 RAG 주입 컨텍스트가 없고 골격 품질이 떨어진다. 세계관 정립 게이트·템플릿 가이드·예시 채움 모드로 완화하나 첫 작품 첫 막의 품질 리스크는 남는다.
- 비트 분해 비용·지연 — 점진 생성·프롬프트 캐싱·모델 다운그레이드로 통제하지만, 한 막에 회차가 많거나 장편이 누적되면 토큰·지연이 여전히 비선형으로 커질 수 있어 회차 수 상한·배치 운영이 필요하다.
- 한국어 웹소설 톤 미반영 — 범용 서사 템플릿(영웅의 여정 등)이 끊기 훅·사이다·회귀물 클리셰 등 웹소설 회차 감각과 어긋나 골격이 '작법서'처럼 나올 위험. stage 가이드·골든셋 튜닝으로 줄이나 1장르 데이터만으로는 한계가 있다.
- 채택률 지표의 잔존 함정 — divergence·수정 깊이 교차 지표를 도입했어도, 작가가 골격을 형식적으로 확정하고 본문에서 사실상 새로 쓰는 패턴을 정량으로 완전히 구분하기는 어렵다. 정성 피드백 의존도가 높게 남는다.
- 온톨로지 cached_entity_state 무효화 — 점검 성능을 위해 캐시한 인물 상태가 세계관 도메인의 진실원과 어긋나면 결정론 경고조차 오탐을 낼 수 있어, 캐시 무효화·스냅샷 버전 정책의 정확성이 신뢰의 전제로 남는다.

### 2.4 작가 워크플로우 & 에디터 통합

**합의된 접근:** 이 도메인은 '가설을 공정하게 측정하고 작가 통제권을 보존하는 그릇'으로 책무를 명확히 한정한다(가설 성패는 LLM 한국어 서사 품질·RAG 정합성 등 타 도메인 소관). P0는 4개 축 + 통합 동선으로 압축: (1) ProseMirror/TipTap 기반 제한 리치텍스트 에디터와 스팬 단위 origin mark(3-state) 추적, (2) 골격→장면 비트 단위 순차 스트리밍 초안 생성과 채택/재생성/폐기, (3) 확정→RagIndexUnit(versionId 키잉·is_active 단일 활성) 멱등 재색인의 가역적 단방향 루프, (4) IndexedDB 로컬 미러+doc_rev 낙관적 동시성+불변 스냅샷의 원고 내구성. SSOT는 PostgreSQL(+pgvector)이며 RAG/온톨로지는 파생 뷰로, 이 도메인은 retrieve/upsertIndex·invalidate와 read-only entity 조회 두 계약만 소비한다. 핵심지표(원형유지율)는 origin mark가 아니라 '확정본 vs AI원안 문자 유사도(LCS)'로 분리 측정하고, 채택률은 구간별로 나눠 콜드스타트와 정상상태를 구분하며, 정량+정성(3~5명 파일럿) 병행을 N=1 한계 보정으로 명문화한다. 그래프DB·외부 벡터DB·5,500자 단발 생성·정밀 블록추적은 기각/보류하고, 설정 참조 핀·진척 보드·버전 복원 UI·연속 동선·인라인 어시스트·export는 P1/P2로 미룬다. 통합 워크스페이스 트리는 '통합 동선' 프레이밍 보존을 위해 최소 구현으로 P0에 남긴다. 스택: Next.js+TS / TipTap(ProseMirror) / FastAPI(Python, AI 도메인 언어 정합) / PostgreSQL+pgvector / 한국어 강세 임베딩(BGE-m3 또는 OpenAI/Cohere) / Claude 주력+모델 추상화 레이어 / 얇은 자체 RAG 파이프라인(LlamaIndex 선택적) / OpenTelemetry+생성 이벤트 계측+LLM 트레이싱.

| origin 추적 정밀도와 핵심지표 측정 방식 | origin은 ProseMirror mark 기반으로 '스팬(문장/범위) 단위'만 추적하고 상태는 3분(ai_generated / author_written / ai_then_edited)으로 두되, 작가 편집 시 ai_then_edited 회색지대를 명시(이분법 강요 금지). 핵심지표 '원형 유지율'은 블록 origin이 아니라 '확정본 vs AI 원안의 문자 단위 LCS/유사도'로 분리 측정. 측정 정의는 MVP 초반에 못 박고 변경 금지. | 정밀 블록 추적에 과투자하지 말고 측정은 유사도로 분리하라는 개발 반론 수용. origin 마크는 '작가 통제권 보존(AI/내 글 구분 표시)'이라는 제품 가치(UX)를 위해 유지하되, 지표 산출의 신뢰성은 마크와 디커플링해 더 싸고 정직하게 확보한다. contenteditable 직접 구현은 금지(범위 추적 불가). |
| AI 생성 단위 (회차 전체 5,500자 vs 장면 비트) | '장면 비트 단위 순차 생성'으로 고정. 한 번에 5,500자 단발 생성 옵션은 기각. 채택률/원형유지율 등 핵심지표의 측정 단위도 '장면'으로 정의함을 합의. | 긴 단발 생성은 후반 설정 일탈·반복·페이싱 붕괴로 채택률을 떨어뜨려 가설을 부당하게 기각시킨다는 개발 반론 수용. 장면 분할 생성은 각 호출 컨텍스트가 짧아 정합성↑·재생성 비용↓. 측정 단위 변경(회차→장면)을 명시적으로 받아들인다. |
| 확정(finalized) 상태의 가역성 (openQuestion → 설계 원칙) | openQuestion에서 MVP 설계 원칙으로 확정: '확정은 가역적이되, RAG는 항상 최신 활성 버전(is_active=true & finalized)만 본다.' RagIndexUnit을 chapterVersionId로 키잉, 확정 회차 수정 시 새 ChapterVersion 생성→옛 unit is_active=false→새 unit 재색인(멱등). PostgreSQL 트랜잭션 + 비동기 잡 멱등화. | 확정 후 수정→RAG 재반영 롤백은 일관성 가설(제품 존재 이유)의 정합성 그 자체이므로 질문으로 미룰 수 없다는 개발 반론 수용. 이중 사실 검색으로 인한 환각/모순 생성을 원천 차단. |
| 리치텍스트 vs 마크다운/플레인 (openQuestion) | '제한된 리치텍스트'로 종결: ProseMirror 스키마를 문단·강조·구분선 등 웹소설 최소 서식으로 좁히고, export는 ProseMirror JSON→마크다운/플레인의 결정적(deterministic) 시리얼라이저로 변환. | 자유 서식 허용은 export 충돌과 origin 추적 복잡도만 키운다는 개발 반론 수용. 웹소설은 서식이 단순하다는 도메인 특성과도 부합. |
| 그래프DB(Neo4j) 도입 여부 | MVP 도입 보류. 온톨로지는 PostgreSQL 엣지 테이블로 시작하고, 이 도메인은 온톨로지의 'read-only entity 조회 API'만 소비. 관계 질의·모순 추론이 본격화되면 그때 Neo4j 검토(온톨로지 도메인 팀 최종 결정). | 단일 작품 수백 노드 규모에서 별도 그래프DB는 콜드스타트 시점 과투자라는 개발 반론 수용. 이 도메인은 온톨로지의 소비자이지 소유자가 아니다. |
| 벡터DB 선택 (pgvector vs 외부 전용DB) | MVP는 pgvector(PostgreSQL 확장) 채택. 본문 권위 저장소와 같은 트랜잭션에서 인덱싱 멱등성·확정 롤백 정합성 보장. 규모 확대 시 Qdrant 등으로 이전. | 단일 작가·수백 회차 규모에 외부 벡터DB(Pinecone/Qdrant)는 시기상조이며 'PG 본문↔외부 인덱스' 이중 정합성 부담만 늘린다는 개발 반론 수용. SSOT는 PostgreSQL, RAG/온톨로지는 파생 뷰. |
| 성공지표의 통계적 한계 (N=1) | 정량(원형유지율=문자 유사도) + 정성(인터뷰) 병행을 MVP 설계에 명문화. 단일 작가 N=1 한계를 인정하고 '가설 기각이 아닌 학습' 프레임으로 운영. 최소 3~5명 파일럿 작가 확보를 온보딩 계획에 포함. 채택률은 '회차 누적 구간별(초반/중반/장기)'로 분리 측정. | N=1 표본으로 '시간 단축'·'다시쓰기 vs 다듬기' 정량 단독 입증은 불가능하다는 개발 반론 수용. 콜드스타트 구간과 정상상태를 혼동하지 않도록 구간 분리 측정도 함께 채택. |
| 이 도메인의 책무 정의 (가설 성패 vs 측정 그릇) | 이 도메인의 1급 산출물은 'origin 추적 + 이벤트 계측(instrumentation) + 원고 내구성'으로 명확히 한정. 채택률을 결정하는 본질(LLM 한국어 서사 품질·RAG 정합성)은 다른 도메인 소관임을 합의하고, 에디터 화려함이 아니라 '품질을 공정하게 측정하고 작가 통제권을 보존하는 그릇'에 자원을 집중. GenerationEvent 이벤트 스키마는 MVP 초기부터 박는다. | 이 도메인이 가설 성패를 만든다고 착각하지 말라는 개발의 냉정한 핵심 판단 수용. 다만 '통제권 있는 단방향 루프'와 '통합 동선'이라는 제품 프레이밍을 그릇 안에서 보존. |
| 온톨로지/RAG 도메인과의 책임 경계 | 이 도메인은 '트리거·이벤트 발행'만 담당. 확정 본문에서 추출된 사실/상태 변화는 온톨로지에 '제안'으로 전달(자동 확정 금지). 의존 API는 두 개로 고정: (a) retrieve(chapterId, skeleton)→컨텍스트 청크+근거, (b) upsertIndex(chapterVersionId)/invalidate(prevVersionId). 온톨로지는 read-only entity 조회 + (P1) 모순감지 결과 수신. API contract를 초기에 합의. | 회색지대(확정 후 온톨로지 자동 갱신)에서 진실 갱신 주체가 불명확하면 정합성 사고가 난다는 개발 반론 수용. 자동 갱신 금지, 제안→검토 루프 원칙. |

**MVP 스코프**
- 단일 작품 워크스페이스 + 좌측 위계 네비게이터(세계관/설정-스토리라인(막·장)-회차 트리) [P0, 최소 구현]
- 회차 집필 에디터: ProseMirror/TipTap 기반 제한된 리치텍스트(문단·강조·구분선), 자수 표시, 자동저장, 회차 메타(목표·등장인물·핵심 사건) 패널 [P0]
- AI 회차 초안 생성 호출: 골격 트리거 → 온톨로지 설정 + RAG 직전/관련 확정 청크 컨텍스트 조립 → '장면 비트 단위 순차' 스트리밍(SSE) 생성 → 제안 블록 삽입 [P0]
- AI 생성분/작가 작성분 origin 표시(ProseMirror mark, 스팬 단위 3-state) + 채택/재생성/폐기 액션 [P0]
- 회차 확정 → RagIndexUnit(versionId 키잉, is_active 단일 활성) 멱등 재색인 단방향 루프 + 가역성(확정 후 수정 시 재색인/비활성화) [P0]
- 원고 손실 방지: IndexedDB 로컬 미러 + doc_rev 낙관적 동시성 서버 저장 + 확정 시 불변 스냅샷 + 저장상태 가시화 + 멀티탭 충돌 감지·경고·머지선택 [P0, openQuestion에서 격상]
- 성공지표 계측: GenerationEvent 이벤트 스키마(채택/폐기/재생성/복원, 원형 생존 글자수) 초기 박기 + 원형유지율=문자 유사도 측정 + 구간별(초반/중반/장기) 채택률 산출 배치 [P0]
- 사이드 설정 참조: 읽기 전용 인라인 조회 최소 수준만 에디터에 흡수(핀/끌어쓰기는 P1) [P0 얇게]
- 온톨로지/RAG 의존 API contract 합의: retrieve / upsertIndex·invalidate + read-only entity 조회 [P0, 인터페이스]

**후속/제외(deferred)**
- 사이드 설정 참조 패널의 핀(고정)·인라인 끌어쓰기 [P1, MVP는 읽기 조회만]
- 회차 상태/연재 진척 보드 [P1로 강등]
- 회차 버전 히스토리 & 되돌리기(스냅샷 목록+복원 UI) [P1] — 단, 확정 시 불변 스냅샷 저장 자체는 P0 내구성에 포함
- 골격→다음 회차 연속 집필 동선 강화 [P1]
- 인라인 부분 재생성·문장 다듬기 어시스트 [P2, 본문 품질 고도화 영역]
- 외부 플랫폼용 내보내기 포맷 [P2] — export 시리얼라이저 설계 원칙만 미리 확정, 기능 구현은 후속
- 그래프DB(Neo4j) 도입 [규모 확대 시 온톨로지 도메인 결정]
- 외부 전용 벡터DB(Pinecone/Qdrant) 이전 [규모 확대 시]
- 실시간 CRDT/OT 다인 동시 편집 [비목표 유지, 낙관적 동시성으로 충돌 감지만]
- 한 번에 5,500자 회차 전체 단발 생성 옵션 [기각]
- 정밀 블록 단위 origin 추적 강화 [기각 — 스팬 단위 + 유사도 측정으로 대체]

**개발 반론(pushback) 요지**
- P0가 너무 많다(6개). 단일 가설 검증 MVP에 P0 6개는 MVP가 아니다. '사이드 설정 참조 패널(읽기+핀)'은 P1로, '회차 상태/진척 보드'도 P1 이하로 내려라. 진짜 P0는 (a)에디터+origin추적, (b)골격→초안 생성, (c)확정→RAG반영 루프, (d)원고 손실 방지 4개뿐이다.
- 원고 손실 방지가 openQuestion에 묻혀 있는데, 이건 질문이 아니라 P0 요구사항이다. 마감 작가에게 원고 유실은 단 한 번으로 제품을 버리게 만든다. 자동저장 '주기'를 고민할 게 아니라 로컬 내구성(IndexedDB)+불변 스냅샷을 MVP 필수로 박아야 한다.
- 'AI생성/작가작성/AI후작가수정' 3분 origin 추적은 정밀도를 높일수록 UX가 무거워지고, 핵심지표(원형유지율)는 굳이 블록 origin이 아니어도 '확정본 vs AI원안의 문자 유사도'로 더 싸고 정직하게 측정된다. 정밀 블록추적에 과투자하지 말고 측정은 유사도로 분리하라.
- 회차 본문을 한 번에 5,500자 생성하는 옵션(openQuestion)은 기각 권고. 긴 단발 생성은 후반 설정 일탈·반복으로 채택률을 떨어뜨려 가설을 부당하게 기각시킨다. '장면 비트 단위 순차 생성'으로 고정하라. 단 채택률 측정 단위도 장면이 됨을 받아들여야 한다.
- 그래프DB(Neo4j 등)는 이 도메인 MVP에 불필요. 단일 작품 수백 노드 규모에서 온톨로지는 PG 엣지 테이블로 충분하고, 이 도메인은 entity read API만 쓰면 된다. 별도 그래프DB 운영은 콜드스타트 시점의 과투자.
- 외부 벡터DB(Pinecone/Qdrant) 분리도 시기상조. pgvector로 본문 권위 저장소와 같은 트랜잭션에서 인덱싱 멱등성·확정 롤백 정합성을 보장하는 편이 훨씬 안전하다. '이중 정합성'(PG↔외부인덱스)을 MVP에 떠안지 마라.
- 성공지표 '회차 1편 확정 시간 단축', '다시쓰기 vs 다듬기 비율'은 단일 작가·단일 작품(N=1) 표본으로는 통계적 결론을 못 낸다. 정량 단독으로 가설 입증을 주장하지 말고 정성 인터뷰 + 최소 3~5명 파일럿을 처음부터 설계에 넣어라. 안 그러면 '측정은 했는데 의미를 모르는' 결과가 나온다.
- '확정' 상태의 가역성을 openQuestion으로 두면 안 된다. 확정 후 수정→RAG 재반영 롤백은 일관성 가설의 정합성 그 자체다. version 핀 + is_active 인덱스 모델로 '확정은 가역, 단 RAG는 항상 최신 활성버전만 본다'를 MVP 설계 원칙으로 먼저 확정하라.
- 리치텍스트 vs 마크다운 논쟁은 '제한된 리치텍스트(ProseMirror 스키마를 문단·강조·구분선으로 좁힘) + 결정적 export 시리얼라이저'로 종결하라. 자유 서식 허용은 export 충돌과 origin 추적 복잡도만 키운다.
- 이 도메인이 가설의 성패를 만든다고 착각하지 마라. 채택률을 결정하는 건 LLM 한국어 서사 품질과 RAG 정합성(다른 도메인)이다. 이 도메인의 진짜 책무는 '품질을 공정하게 측정하고 작가 통제권을 보존하는 그릇'을 만드는 것. 에디터 화려함이 아니라 origin 추적·계측·내구성에 자원을 몰아라.

**남은 리스크**
- 콜드스타트: 1~2회차 시점엔 참조할 확정 회차가 거의 없어 RAG 기여가 낮고 초반 채택률이 낮게 측정될 위험. 완화(초반 온톨로지 비중↑·구간별 측정)는 하지만 초기 인상이 가설을 부당하게 기각시킬 잔존 위험이 남는다.
- 가설 성패가 이 도메인 통제 밖(LLM 한국어 서사 품질·RAG 정합성)에 있다는 구조적 한계. 측정 그릇이 완벽해도 타 도메인 품질이 낮으면 채택률은 낮게 나온다.
- N=1 표본의 통계적 무의미성. 3~5명 파일럿을 확보하더라도 단일 장르·단일 작품 표본으로는 '시간 단축'·'다시쓰기 vs 다듬기' 결론의 일반화가 어렵고, 정성 해석에 의존하는 위험이 남는다.
- origin 추적의 본질적 모호성: 작가가 AI 텍스트를 이동·병합·재배열하면 스팬 경계가 흐려져 ai_then_edited 회색지대가 비대해질 수 있음. 유사도 측정으로 지표는 보호되나 화면상 'AI/내 글 구분'의 정확도는 저하될 잔존 위험.
- 온톨로지/RAG 도메인과의 API contract가 초기 합의대로 병렬 제공되지 않으면 W9~10 확정→인덱싱 파이프라인 일정이 직접 막힘(의존성 리스크).
- 비용/지연: 장면 단위 순차 생성은 정합성을 높이지만 호출 횟수가 늘어 토큰·지연 누적 가능. 요약 캐시·프롬프트 캐싱·토큰 예산 캡으로 완화하나 마감 작가 체감 속도와의 균형은 실측 검증 필요.
- 확정 가역성의 운영 복잡도: 멱등 재색인·is_active 전환이 비동기 잡 실패/지연 시 일시적으로 옛 본문이 검색에 노출될 윈도가 존재(트랜잭션 경계와 잡 큐 신뢰성에 의존).
- MVP 단일 장르 확정 미해결(현대 판타지 가정): 에디터/골격 UX 도메인 디폴트에 영향을 주므로 기획 측 조기 확정 필요. 미확정 시 일부 UX 결정이 지연될 잔존 리스크.

## 3. 비평 회의 (적대적 검증)

### 3.1 기획 비평 — planning-critic

**총평:** 한 줄 평: 기술적으로는 흠잡을 데 없이 합의된 PRD지만, 세 도메인 모두 같은 SWE 사각지대를 공유한 채 "결정성·정합성·측정"에만 합의해서, 정작 제품의 생사를 가르는 비기술 질문(검증할 작가를 어떻게 구하나, 누가 왜 돈을 내나, 작가 원고를 RAG에 넣는 게 법적으로/신뢰상 되나, 가설 통과/사망 기준 숫자가 뭔가)이 통째로 빠진 "정교한 기능 명세서"다. 출시 가능 여부를 묻기 전에, 이 기획은 "검증 가능 여부"부터 증명하지 못했다. // 가장 치명적인 진실: MVP의 단 하나의 가설은 RAG+일관성검증 공동책임으로 떠넘겨졌고, 세 번째·네 번째 도메인(스토리구조·에디터)은 스스로 "나는 가설 성패를 통제하지 못한다"고 인정했다. 그렇다면 일관성 가설을 실제로 책임지는 건 RAG+온톨로지 두 도메인뿐인데, 그 가설을 증명할 표본(기성 작가의 수백 회차 원고)을 어떻게 합법적으로 확보하는지가 remainingRisks 한 줄로만 적혀 있다. 검증 표본 조달이 안 되면 나머지 모든 정교한 합의는 작동하지 않는 엔진을 위한 완벽한 차체다. // 진행 권고: 코딩 시작 전 2~3주짜리 "검증 가능성 스파이크"로 (1) 협력 기성 작가 3~5명 + 임포트 동의/저작권 처리, (2) 골든셋 50~100건 실제 라벨링 가능성, (3) recall@k 게이트 통과 여부를 먼저 증명하라. 이 셋 중 하나라도 막히면 8~12주 P0는 의미가 없다.

| 심각도 | 우려 | 이유 | 권고 |
|---|---|---|---|
| critical | MVP를 검증할 '기성 작가의 수백 회차 원고'를 어떻게 합법적으로 조달하는지가 전부 remainingRisks 한 줄로 처리됐다. 검증 시나리오를 '기성 작품 임포트'로 공식 채택해놓고, 그 핵심 전제인 '협력 작가 확보 + 미공개 원고 대량 임포트의 저작권/개인정보 동의'는 합의 대상조차 아니었다. | RAG 도메인이 명시했듯 백지 작품으로는 인덱스가 비어 RAG ON/OFF 차이가 없어 MVP 기간 내 가설 검증이 '불가능'하다. 즉 임포트 시나리오가 막히면 제품의 단 하나의 가설을 검증할 방법 자체가 사라진다. 그런데 (1) 연재 중인 기성 작가가 미완결 작품의 전체 원고를 외부 AI 툴에 통째로 올릴 인센티브가 거의 없고(유출·표절 학습 공포가 이 직군의 1번 공포다), (2) 한국 저작권법상 원고 인덱싱·임베딩·재생성 활용은 '내 글로 너희가 학습하느냐'는 신뢰 질문과 개인정보처리방침·이용약관의 데이터 사용 조항을 P0로 요구한다. 온프레미스 임베딩으로 '보호 가능한지 실무 확인 필요'라고만 적은 건 출시 직전 터질 폭탄이다. | 코딩 1주차가 아니라 코딩 시작 '전'에 검증 가능성 스파이크를 둬라. (a) 임포트 동의서·데이터 사용 약관·삭제권 보장을 갖춘 협력 작가 3~5명을 실제로 서면 확보했는가를 게이트로 삼고, 확보 못 하면 P0를 시작하지 마라. (b) '작가 원고는 학습에 쓰지 않고 해당 작가 워크스페이스 내 검색에만 쓴다'는 데이터 격리 보증을 제품 1급 약속으로 명문화하고 기술 아키텍처(테넌트 격리, 모델 파인튜닝 금지)로 못 박아라. (c) 골든셋 50~100건 라벨링은 작가 부담이 큰 작업인데, 그 작업을 누가 어떤 보상으로 할지가 채택 작가 확보와 한 묶음이다. |
| critical | 수익 모델·단위경제·과금 트리거가 세 도메인 어디에도 없다. 가격이 없는 게 아니라 '누가 왜 돈을 내는가'라는 질문 자체가 PRD에 존재하지 않는다. | 이건 LLM 생성 SaaS다. 회차당 5,500자 생성 + 직전 N회차 요약 + 검색 컨텍스트 + Sonnet/GPT-4.1급 생성을 매일 5~7회, 수백 회차 누적으로 돌린다. 토큰 변동비가 사용량에 정비례하는 구조에서 단위경제는 출시 후 최적화 과제가 아니라 출시 전 생존 조건이다(프롬프트 캐싱·요약·슬롯 cap을 P0에 넣은 것 자체가 비용이 무섭다는 증거다). 더 근본적으로, 이 제품의 타깃인 '마감에 쫓기는 기성 웹소설 작가'의 실질 수입 분포는 양극화가 극심하다. 상위 소수를 빼면 월 수십만 원 수준 작가가 다수인데, 이들이 월 구독료로 감당할 수 있는 금액과 작가 1인당 LLM 변동비가 교차하는지(즉 ARPU > 변동비 + CAC)가 검증 안 되면, 가설이 다 맞아도 사업이 적자다. mvpThesis는 '다듬어 쓸 가치를 느낀다'까지만 증명하는데, '가치를 느끼는 것'과 '그 가치에 변동비 이상을 지불하는 것'은 별개다. | MVP 가설에 '지불 의사' 축을 추가하라. 정성 인터뷰에서 '이 초안 품질이면 월 얼마까지 내겠는가'를 회차 작업마다 측정하고, 작가 1인당 월 LLM 변동비(생성+요약+임베딩+재인덱싱)를 P0 기간에 실측해 단위경제 손익분기 구독가를 산출하라. 그 가격이 타깃 작가 다수의 지불 의사를 넘으면 페르소나를 '상위 연재 작가'로 좁혀야 하고, 그건 페르소나 3개 정의 자체를 다시 써야 한다는 뜻이다. |
| critical | '일관성 가설'의 책임이 도메인 간에 무한히 핑퐁된다. 에디터는 'RAG/온톨로지 소관', 스토리구조는 'RAG 품질 종속', RAG는 '일관성검증과 공동가설'이라고 각자 책임을 옆으로 미뤘다. 결과적으로 누구도 '우리가 이 숫자를 이만큼 올린다'를 단독으로 약속하지 않는다. | 메모리의 PMF 임계치 미정의 패턴 그대로다. mvpThesis는 '작가가 처음부터 다시 쓰지 않고 다듬어 쓸 가치를 느낀다'인데, 이 공동 가설을 '성공'이라 부를 임계 숫자가 어디에도 없다. '재작성 비율 감소 = 다수 우위', '원형 유지율'을 측정하겠다는 말은 있지만, 원형 유지율 몇 %부터 성공이고 몇 % 미만이면 이 제품을 접는가(kill criteria)가 없다. 책임이 세 도메인에 분산되고 임계치가 없으면, 3개월 후 결과는 반드시 '잘 모르겠다, 도메인별로 더 봐야 한다'가 된다. 이건 가설 검증이 아니라 가설 검증의 무한 연기다. N=1~5 표본의 신뢰구간이 넓다는 걸 스스로 인정한 상태라 더 위험하다. | P0 착수 전에 '공동 가설'의 단일 합격선을 숫자로 못 박아라. 예: '파일럿 작가 5명 중 4명이 RAG ON 초안의 원형 유지율 X% 이상 + 블라인드 비교에서 RAG ON을 Y회 중 Z회 이상 설정 일치로 선택'. 동시에 kill criteria도 명시하라: '원형 유지율 중앙값이 W% 미만이면 RAG 접근을 폐기'. 임계치 없는 측정 인프라는 정직해 보이는 미루기다. |
| high | 세 도메인이 전부 '도메인 간 API 계약을 선행 작업으로 동결'에 의존하는데, 계약의 주체·기한·소유자가 없다. 모두가 '상대가 read-only API를 제때 줄 것'을 전제로 자기 일정을 짰다. | 에디터는 retrieve/upsertIndex·invalidate 두 계약에 의존하고, 스토리구조는 온톨로지·RAG의 읽기 API와 확정 이벤트에 의존하며, RAG는 설정 도메인의 엔티티ID 단독 발급에 의존하고, 온톨로지는 RAG가 chapter_id를 소유한다고 가정한다. 즉 네 개의 도메인이 서로를 가리키는 순환 의존이다. 각 도메인 remainingRisks에 '계약 미합의 시 +2~3주', '일정 직접 막힘'이 적혀 있는데, 이 버퍼들이 독립적으로 더해지면 실제 슬립은 합산이 아니라 곱셈에 가깝다(한 계약이 늦으면 그에 의존한 세 도메인이 동시에 막힌다). PRD에는 이 통합을 책임지는 단일 오너나 계약 동결 데드라인이 없다. | 코딩 전에 '도메인 간 인터페이스 계약'을 단일 문서로, 단일 오너(통합 아키텍트)가, 명시적 데드라인 내에 동결하는 것을 0순위 마일스톤으로 둬라. 특히 엔티티ID 발급 주체(설정 도메인 단독), chapter_id 소유(RAG), 충돌 판정 책임(일관성검증)의 경계는 코드 한 줄 쓰기 전에 OpenAPI 스텁으로 고정하고 mock으로 병렬 개발하라. 계약이 안 잡히면 P0 일정 자체가 허구다. |
| high | 페르소나 3개가 한 제품의 P0를 정당화하는 데 쓰이지만, 콜드스타트 구조상 '신인 작가'는 MVP에서 사실상 버려졌다. 그런데 차별점 광고는 신인의 '중반 붕괴 방지'를 핵심으로 내세운다. | 검증 시나리오를 '수백 회차 기성 작가 임포트'로 고정한 순간, 온톨로지·RAG·이전 회차가 빈약한 신인 작가는 RAG 주입 컨텍스트가 없어 골격 품질이 떨어진다(스토리구조·에디터 도메인이 직접 인정). 즉 MVP가 실제로 가치를 증명하는 대상은 페르소나 1·3(기성·복잡설정 작가)뿐이고, 페르소나 2(신인)는 '완화하나 리스크 남음'으로 처리됐다. 그런데 differentiators의 '구조 우선 설계가 신인의 중반 붕괴를 구조적으로 방지'와 스토리구조 도메인의 존재 이유 절반이 신인을 향한다. 검증되는 사용자와 광고하는 사용자가 어긋난 페르소나 미스핏이다. 더구나 기성 작가는 이미 자기 워크플로우(한글, 메모장, 노션)가 굳어 있어 전환 비용이 높고, 진짜 경쟁자는 '아무것도 안 바꾸기'다. | MVP의 검증 대상 페르소나를 1명으로 좁혀 명시하라(기성 또는 복잡설정 작가 중 하나). 신인 작가 가치와 스토리구조의 '중반 붕괴 방지'는 MVP 가설에서 명시적으로 제외하고, 검증 후 별도 가설로 분리하라. 동시에 '기성 작가가 굳은 워크플로우를 버리고 이걸 쓸 전환 트리거'가 무엇인지를 인터뷰로 검증하라 — 설정 충돌 댓글 지적의 빈도·고통이 도구 전환 비용을 넘는지가 진짜 must-have 판별 기준이다. |
| high | 범위가 여전히 과욕이다. '단 하나의 가설'을 검증한다면서 P0에 4개 도메인 × 각 10여 개 P0 항목 + 시점모델·아웃박스·멱등워커·골든셋 게이트·블라인드 A/B·구간별 측정·origin 추적·로컬 내구성까지 들어가 있다. 8~12주 + 1~2명 규모로 불가능하다. | 메모리의 sprint 과부하 패턴이다. 각 도메인이 개별적으로는 '최소'라고 주장하지만, 합치면 거대하다. 진짜 가설(온톨로지+RAG 주입이 일관성 체감을 올리는가)을 검증하는 데 필요한 최소는 사실 RAG+온톨로지 두 도메인의 일부다. 에디터의 ProseMirror origin 마크, 멀티탭 충돌 머지, 스토리구조의 떡밥 수동 태깅·재생성 안전장치 낙관적 락 등은 전부 '있으면 좋지만 가설 검증과 무관한' 비타민인데 P0로 들어왔다. 특히 시점모델(EntityStateTimeline)을 데이터·판정 레벨 P0로 승격한 결정은 기술적으로는 옳지만, 단일 작가·단일 작품으로 일관성 '체감'만 검증하는 단계에서 회상/시간역행 서사의 오탐을 잡겠다고 시점 인지 판정 로직을 처음부터 만드는 건 검증 전 과투자일 수 있다. | '가설 검증에 필수인가'를 단일 기준으로 P0를 재절단하라. 진짜 P0는 (1) 기성 원고 임포트·인덱싱, (2) 온톨로지 정형 필드 결정론 주입 + RAG 검색 주입, (3) 골격/초안 생성, (4) 블라인드 A/B 측정 인프라, (5) 원고 손실 방지. 시점모델·아웃박스·origin 마크·멱등 재인덱싱·떡밥 태깅은 '검증 후 확장'으로 미뤄도 가설은 검증된다. 측정 인프라만큼은 P0를 사수하되, 신뢰성 엔지니어링(아웃박스/멱등)은 파일럿 5명 규모에선 수동 운영으로 대체 가능한지 따져라. |
| high | '주입 ≠ 준수'를 핵심 리스크로 인정해놓고, 그 리스크의 해결책을 'constraint 블록·캐시 고정 주입으로 완화'라는 프롬프트 기법으로만 처리했다. 제품의 단 하나의 가설이 LLM의 지시 준수율이라는 통제 불가능한 변수에 걸려 있다. | RAG가 정확한 설정(눈 색깔=청색)을 프롬프트에 완벽히 박아도, 생성 LLM이 그걸 무시하거나 그럴듯한 새 설정을 환각하면 일관성은 무너진다. 이 제품의 모든 정교한 결정론 인프라(온톨로지 SSOT, 시점모델, 골든셋)는 '검색·주입의 정확성'을 보장할 뿐, '생성 결과의 준수'를 보장하지 못한다. 그런데 mvpThesis가 약속하는 건 후자다('AI가 내 세계관을 어기지 않는다'). RAG remainingRisks에 '준수율이 작가 체감 임계치에 못 미칠 가능성'이 잔존 위험으로 적혀 있는데, 이건 잔존 위험이 아니라 가설의 본체다. 한국어 웹소설 문체·고유명사에서 Sonnet/GPT급 모델의 설정 준수율이 실제로 작가가 '다시 안 쓰고 다듬어 쓸' 수준인지는 아무도 모른다. | P0 1~2주차에 '주입 후 준수율'을 직접 측정하는 미니 실험을 recall@k 게이트와 동급으로 둬라. 완벽한 설정을 박은 프롬프트로 한국어 회차 초안을 N개 생성해, 작가가 '설정 충돌로 다시 써야 함'으로 판정한 비율을 측정하라. 이 준수율이 임계치 미만이면 RAG 품질을 아무리 올려도 가설은 실패다 — 그럼 검색 인프라에 자원을 더 붓기 전에 생성 단 제어(설정 위반 자동 재생성, 후처리 검증 루프)로 전략을 틀어야 한다. |

**사각지대(blind spots)**
- 경쟁/대체재 분석이 ChatGPT·Scrivener 비교에 머문다. 진짜 경쟁자는 (a) '아무것도 안 바꾸기'(기성 작가의 굳은 한글+메모장 워크플로우)와 (b) 작가가 이미 ChatGPT에 자기 설정을 매번 붙여넣는 '무료지만 귀찮은' 우회법이다. 이 우회법의 고통이 도구 전환·구독 비용을 넘는다는 증거가 없다.
- 어뷰징/악용 시나리오가 전무하다. AI 생성 SaaS에서 최소 3개는 시뮬레이션해야 한다: (1) 타사 작가의 인기작 전체를 임포트해 '문체·설정을 학습한 표절 초안' 양산, (2) 작가가 자기 워크스페이스에 올린 타인 저작물(번역·2차창작 원작)의 인덱싱, (3) 생성된 초안이 기존 출판작과 우연/의도 일치할 때의 표절 분쟁. 작가 원고를 RAG에 넣는 제품에서 이건 운영·법무의 핵심인데 한 줄도 없다.
- 운영 인건비가 보이지 않는다. 골든셋 50~100건 라벨링, 파일럿 작가 3~5명 온보딩·인터뷰, 충돌 인박스 오탐 튜닝, 임포트 마법사 대응, 작가 원고 삭제권 처리 — 이 모두가 사람이 붙는 일인데 P0 인력 산정(백엔드 1~2 + 프론트 1)에 운영·작가 리에종 인력이 없다.
- AI 생성물의 저작권 귀속과 표시 의무. 한국에서 AI 생성 비중이 큰 콘텐츠를 카카오페이지·네이버시리즈에 송고할 때 플랫폼 정책(AI 사용 고지)과 저작물성 논란이 있다. 이 제품으로 만든 초안이 작가의 저작물로 인정받는지, 플랫폼이 AI 활용작을 제재하는지가 작가의 '다듬어 쓸 가치' 판단에 직결되는데 검토 흔적이 없다.
- MVP 단일 장르를 '현대 판타지'로 가정만 하고 확정하지 않았다. 에디터·골격 UX·온톨로지 스키마·골든셋이 전부 이 결정에 종속되는데 미확정이다. 더 중요한 건, 현대 판타지/이세계물은 '회귀·각성·변신·환생'으로 인물 상태가 의도적으로 격변하는 장르라 결정론 충돌 감지의 오탐이 가장 심한 장르라는 점 — 검증에 가장 불리한 장르를 디폴트로 골랐을 수 있다.
- 측정 도구가 측정 대상을 오염시키는 자기참조 위험. RAG ON/OFF 블라인드 A/B를 제품 자체가 수집·집계하는데, 표본이 N=5 파일럿이고 그 작가들이 '협력 작가'(이미 호의적)라면 결과는 구조적으로 긍정 편향된다. 메모리의 자기참조 ROI 오류 변형이다. 외부 비협력 작가의 무보정 결과 없이는 가설 '입증'은 마케팅 수치다.
- '확정분만 기억' 신뢰 모델과 '미확정 초안 휘발성 단기 참조'의 사용자 멘탈모델 충돌. 작가는 직전에 쓴 미확정 초안이 다음 생성에 반영되길 기대하는데(연속 집필), 영구 인덱싱과 휘발성 주입의 경계는 작가에게 비가시적이다. '왜 방금 쓴 내용을 기억 못 하지?'라는 혼란이 신뢰를 갉는다.
- 데이터 이식성/종료 시나리오. 작가가 수백 회차 원고와 공들여 만든 온톨로지를 이 툴에 쌓은 뒤 서비스를 떠날 때(또는 서비스가 종료될 때) 자기 자산을 온전히 반출할 수 있는가? 창작자에게 lock-in 공포는 도입 장벽이고, 이게 보장 안 되면 기성 작가는 핵심 작품을 절대 안 올린다. 메모리의 'B2B SaaS 반출 누락' 패턴이 B2C 창작툴에도 그대로 적용된다.

### 3.2 아키텍처 비평 — code-critic

**총평:** 개별 도메인 합의는 각자 충분히 self-critique를 거쳐 단독으로는 견고하다. 그러나 이 통합본은 '4개 도메인이 합쳐졌을 때 하나의 시스템으로 작동하는가'를 한 번도 검증하지 않았다 — 바로 거기에 머지 차단급 결함이 몰려 있다. 세 가지가 프로덕션 사망 지점이다: (1) Chapter 엔티티의 소유권을 에디터·RAG가 동시에 주장하고 RagIndexUnit/ContentChunk가 같은 테이블의 두 이름이라, SSOT를 표방하는 제품이 진실 소스를 둘로 쪼개는 split-brain을 내장한다. (2) 회차 확정 이벤트의 3중 컨슈머(RAG/온톨로지/스토리구조)에 부분 실패 보상 설계가 없어, 온톨로지 상태 반영이 RAG 재색인보다 늦으면 일관성 검증이 방금 확정한 본문과 모순된 헛경고를 쏟아내 핵심 차별점(충돌 감지 신뢰)을 스스로 죽인다. (3) MVP의 단 하나의 가설을 4개 도메인이 모두 '내 책임 아님'이라 명시해 증명 책임에 공백이 생겼고, 세 도메인의 A/B 측정 단위(골격/본문/장면)가 어긋나 통합 분석이 불가능하다. 추가로 콜드스타트 검증 시나리오가 도메인마다 다르고('기성 임포트' vs '신규 백지'), 시점 키(chapter_id/chapterNo/seq_no)가 통일 안 됐으며, 회차 1편 생성의 end-to-end 지연 예산을 아무도 합산하지 않았다. 블라인드 스팟으로는 외부 LLM에 미공개 원고를 전송하면서 동의·DPA·파기 정책이 전무한 점, append-only 이력과 개인정보 파기 의무의 정면 충돌, 멀티테넌트 격리·IDOR 미검증, 도메인 간 trace 상관관계 부재가 있다. 결론: 도메인별로는 머지 가능하나, 통합 시스템으로는 머지 불가다. 코드 착수 전에 (a) Chapter 단일 writer 계약, (b) 확정 이벤트 다중 컨슈머 정합성·워터마크, (c) 단일 A/B 프로토콜과 단일 측정 단위 owner, 이 세 가지를 cross-domain 계약 문서로 동결하는 것이 1순위다. 이것 없이 4개 팀이 병렬 개발하면 통합 단계에서 데이터 정합성 사고로 일관성 가설 자체를 검증조차 못 하게 된다.

| Blocker | Chapter 엔티티의 소유권이 세 도메인에서 동시에 주장된다 — 에디터는 'SSOT=PostgreSQL, Chapter/ChapterVersion 권위 저장소를 소유'라 명시하고, RAG는 'chapter_id 소유권은 생성/RAG 도메인'이라 명시한다. 스토리구조는 '확정 이벤트를 발행'한다. 같은 Chapter 레코드에 대해 누가 is_active/finalized 상태의 진실원이며 누가 쓰기 권한을 갖는지가 합의되지 않았다. | 이 모순은 프로덕션에서 정확히 일관성 가설을 깨는 지점에서 터진다. 에디터 합의는 '확정은 가역적이되 RAG는 항상 is_active=true 최신 버전만 본다'를 자기 도메인 안에서 보장한다. 그러나 RAG 합의는 같은 Chapter를 자기가 소유한다고 보고 'isConfirmed 커밋 + 아웃박스'를 자기 트랜잭션으로 관리한다. 두 도메인이 각자 Chapter 상태를 쓰면 (a) 확정 롤백 시 RAG는 옛 버전을, 에디터는 새 버전을 진실로 보는 split-brain이 발생하고, (b) RagIndexUnit(에디터 정의)과 ContentChunk(RAG 정의)는 사실상 같은 테이블의 두 이름인데 키 전략이 versionId 키잉 vs contentHash 증분으로 달라 같은 회차에 두 인덱싱 경로가 동시 작동하면 stale 청크가 영구 잔존한다. SSOT를 표방하는 제품이 진실 소스를 둘로 쪼개는 자기부정이다. | 도메인 간 계약을 코드 착수 전 동결할 때 'Chapter 레코드의 단일 writer'를 명문화하라. 권고: 에디터 도메인이 Chapter/ChapterVersion/finalized 상태의 유일한 writer이고, RAG는 chapter_id를 외래 참조만 하며 절대 Chapter 상태를 쓰지 않는다(RAG가 소유하는 것은 ContentChunk/RagIndexUnit 단일 테이블뿐). RagIndexUnit과 ContentChunk를 단일 스키마로 통합하고 키 전략을 versionId 키잉으로 일원화하라(contentHash는 그 안의 변경감지 보조 컬럼으로만). '회차 확정 이벤트'는 에디터가 단독 발행하고 RAG는 컨슈머다. 이 단일 writer 원칙을 OpenAPI 계약 문서 최상단에 박아라. |
| Blocker | 회차 확정 이벤트의 3중 컨슈머(RAG 재색인 + 온톨로지 상태갱신 제안 + 스토리구조 divergence 계산)에 대한 부분 실패/보상 트랜잭션 설계가 어느 도메인 합의에도 없다. 각 도메인은 자기 outbox+멱등 워커만 다루고, 한 이벤트가 세 소비자 중 일부만 성공한 상태를 누구도 책임지지 않는다. | at-least-once outbox는 각 컨슈머의 멱등성만 보장하지 '세 컨슈머 간 정합성'은 보장하지 않는다. 시나리오: 작가가 회차를 확정한다 → RAG 재색인은 성공해 새 본문이 검색에 노출됨 → 온톨로지 상태갱신 제안 워커는 죽어서 인물 상태(예: 사망)가 EntityStateTimeline에 반영 안 됨. 이제 RAG는 '인물이 죽은 새 본문'을 주입하는데 온톨로지 결정론 충돌 감지는 '인물이 살아있다'는 옛 상태로 판정한다. 즉 일관성 검증 도메인이 자기가 방금 확정시킨 본문과 모순된 경고를 쏟아낸다. 작가는 한두 번 이 헛경고를 보면(온톨로지 합의가 직접 적시한 'ignore 위험') 충돌 인박스 전체를 신뢰하지 않게 되어 MVP의 핵심 차별점이 죽는다. 이것은 단일 도메인 합의로는 절대 발견 못 하는, 도메인 경계에서만 터지는 사고다. | 확정 이벤트의 소비를 '동기 필수 vs 비동기 best-effort'로 명시 분리하라. RAG 재색인은 비동기여도 무방하나, 온톨로지 상태 반영 지연 동안 충돌 감지가 stale 상태로 판정하지 않도록 ConsistencyFlag 판정 시 '해당 회차의 온톨로지 반영 완료 여부'를 게이트로 확인하고 미반영이면 판정 보류(경고를 띄우지 않음)하라. 또는 온톨로지 합의가 이미 가진 entity_revision_refs/EntityStateTimeline의 effective_from_chapter_id를 활용해, 충돌 판정을 'as_of_chapter'가 인덱싱 완료된 회차까지만 수행하도록 워터마크(last_consistent_chapter_no)를 도입하라. 세 컨슈머의 진행 상태를 한곳에서 관측할 수 있는 통합 이벤트 상태 테이블이 필요하다. |
| Blocker | MVP의 단 하나의 가설('온톨로지+RAG 주입으로 일관성이 일반 LLM 대비 체감될 만큼 높아진다')의 증명 책임이 4개 도메인에 흩어져 있고, 어느 도메인도 이를 단독 증명할 수 없다고 각자 명시한다. RAG는 'RAG+일관성검증 공동 가설', 스토리구조는 'RAG 품질에 종속', 에디터는 '측정 그릇일 뿐 가설 성패는 타 도메인'이라 한다. 정작 '일관성 검증 도메인'은 이 통합본에 독립 도메인으로 존재하지 않고 온톨로지 도메인 안에 흡수돼 있다. | 세 도메인이 모두 '일관성 가설은 내 책임이 아니라 옆 도메인 책임'이라고 정확히 같은 말을 한다. 이것은 책임의 명확한 분배가 아니라 책임의 공백이다. 가설을 실제로 증명하는 통합 측정 파이프라인(온톨로지 주입 ON/OFF 블라인드 A/B + 재작성 비율 태깅)은 온톨로지 합의에도, RAG 합의에도 '내장한다'고 적혀 있지만 두 A/B 프로토콜이 같은 것인지 다른 것인지, 누가 운영하는지가 미정이다. 에디터가 GenerationEvent로 원형유지율을 측정하는데, 이 측정 단위는 '장면'이고, 온톨로지 A/B는 '회차 골격', RAG A/B는 '회차 본문' 단위다. 세 측정의 단위가 어긋나 통합 분석이 불가능하다. 3개월 뒤 '측정은 했는데 가설은 잘 모르겠다'가 결론이 될 구조적 위험을 네 도메인이 각자 적어놓고도 누구도 통합 책임을 안 진다. | MVP 착수 전에 '일관성 가설 검증'을 독립된 5번째 워크스트림(또는 명시적 owner를 가진 cross-cutting 책임)으로 분리하고, 단일 A/B 프로토콜과 단일 측정 단위(권고: 회차)를 강제하라. 온톨로지 A/B, RAG A/B, 에디터 GenerationEvent를 하나의 실험 설계로 통합해 동일 회차에 대해 [온톨로지+RAG 주입 / RAG만 / 무주입] 3-arm을 블라인드로 작가에게 제시하고 '다시 쓴 비율'을 단일 정의로 태깅하라. 측정 그릇(에디터)이 측정 단위의 진실원이므로, 온톨로지/RAG의 A/B 정의를 에디터의 GenerationEvent 스키마에 종속시켜라. |
| High | 콜드스타트 우회 전략이 도메인마다 다르고 서로 충돌한다. RAG는 '기성 작가의 수십~수백 회차 작품 임포트'를 가설 검증 1차 시나리오로 공식 채택했다. 그러나 온톨로지는 '단일 작가·단일 작품·신규 백지'를 전제로 시점 입력 마찰을 논하고, 스토리구조는 신인 작가의 '백지에서 막 골격 생성'을 핵심으로 한다. | RAG가 채택한 '기성 작품 임포트' 시나리오는 가설 검증을 가능하게 하지만, 그 시나리오에서는 (a) 온톨로지가 비어 있다 — 수백 회차를 임포트해도 인물·세력·EntityStateTimeline은 자동으로 안 채워진다(온톨로지 합의는 자동 추출을 P2로 미뤘다). 즉 RAG는 임포트로 풍부한데 온톨로지는 콜드 상태라, 결정론 주입(온톨로지)과 RAG 검색 주입의 비대칭이 발생한다. (b) 스토리구조의 막 골격 생성은 '이미 수백 회차 쓴 작품'에는 무의미하다 — 골격은 이미 본문으로 존재한다. 세 도메인이 서로 다른 검증 시나리오를 전제하면, 임포트 시나리오로 RAG를 검증하는 동안 온톨로지와 스토리구조는 검증되지 않고, 백지 시나리오로 스토리구조를 검증하는 동안 RAG는 콜드스타트로 무력하다. 어떤 단일 시나리오로도 4개 도메인을 동시에 검증할 수 없다. | MVP 검증 시나리오를 4개 도메인이 공유하는 단일 정의로 강제 통일하라. 임포트 시나리오를 채택한다면, 임포트 시 온톨로지 엔티티 자동 추출(현재 P2)을 '임포트 한정'으로 P0에 끌어올려야 RAG-온톨로지 비대칭이 해소된다. 또는 스토리구조 검증을 별도 시나리오로 분리하고 '임포트 작품은 골격 생성을 검증 범위에서 제외'한다고 명문화하라. 지금처럼 도메인마다 암묵적으로 다른 페르소나·시나리오를 전제하면 통합 데모 자체가 성립하지 않는다. |
| High | 온톨로지의 EntityStateTimeline.effective_from_chapter_id와 RAG의 chapterNo<=N 메타데이터 필터가 '회차 시점'이라는 같은 개념을 다른 키로 표현한다. 한쪽은 chapter_id(uuid FK), 한쪽은 chapterNo(int). 스토리구조의 ChapterBeat는 또 seq_no를 쓴다. 회차의 시간적 순서를 표현하는 키가 도메인마다 다르다. | as_of_chapter 시점 기준 충돌 판정(온톨로지)과 chapterNo<=현재 검색 필터(RAG)는 둘 다 '이 회차 시점에 진실인 상태/본문만 본다'는 동일 목적이다. 그런데 온톨로지는 chapter_id로, RAG는 chapterNo(정수)로 시점을 표현한다. 회차 삽입/재정렬(스토리구조가 드래그 재정렬을 P0로 가짐)이 일어나면 chapterNo 정수가 재배열되는데, 온톨로지의 effective_from_chapter_id(uuid)는 안정적이라 두 시점 좌표계가 어긋난다. 예: 작가가 5화와 6화 사이에 새 회차를 삽입 → chapterNo 재배열 → RAG는 'chapterNo<=6'으로 옛 6화를 포함하는데 온톨로지는 chapter_id 기준이라 다른 회차를 본다. 시점 정합성이 깨지면 '회상/시간역행 오탐 방지'라는 온톨로지의 핵심 장치가 무력화된다. | '회차 시점'을 표현하는 단일 정렬 키를 4개 도메인 공통 계약으로 정의하라. chapterNo(재배열 가능한 정수)를 시점 판정에 쓰지 말고, 안정적 chapter_id + 별도의 'narrative_order'(재배열 시에만 갱신되는 fractional index 또는 lexorank) 조합으로 통일하라. 시점 비교는 narrative_order로, 식별은 chapter_id로 분리한다. 재정렬이 시점 좌표계를 깨지 않도록 narrative_order 변경 시 온톨로지 EntityStateTimeline과 RAG 메타데이터를 함께 갱신하는 책임을 명시하라. |
| High | 비용·지연이 4개 도메인 각각에서 '통제한다'고 적혀 있으나, 단일 회차 생성 1회에 누적되는 LLM 호출 체인의 end-to-end p95가 어디에도 합산되지 않았다. 스토리구조(쿼리 재작성+골격), RAG(쿼리 재작성+검색+생성), 에디터(장면 비트 단위 순차 생성)가 각자 LLM을 호출한다. | 에디터 합의는 '장면 비트 단위 순차 생성'을 확정했다. 한 회차가 N개 장면이면 회차당 N회 LLM 생성 호출이다. 각 호출 전에 RAG 컨텍스트 빌더가 하이브리드 검색(벡터+BM25 RRF)을 돌리고, 스토리구조는 별도로 쿼리 재작성 LLM을 호출한다. 5,500자 회차를 5개 장면으로 나누면 회차 1편당 최소 5회 생성 + 5회 검색 + 보조 LLM 호출이 직렬로 쌓인다. 마감에 쫓기는 작가(핵심 페르소나)에게 회차 1편 생성이 누적 지연으로 수 분이 걸리면 '속도' 니즈가 정면으로 깨진다. 각 도메인은 자기 호출의 p95만 보고 'Med 리스크, 캐싱으로 완화'라 적었지만, 직렬 체인의 합산 p95는 아무도 측정하지 않는다. 숫자 없는 '캐싱으로 완화'는 미신이다. | 회차 1편 생성의 end-to-end 지연 예산(latency budget)을 cross-cutting 요구사항으로 명시하고 슬롯별로 배분하라(예: 회차당 총 90초, 장면당 15초, 그 안에서 검색 2초/생성 12초/보조 1초). 장면 비트 순차 생성을 무조건 직렬로 하지 말고, 장면 간 의존이 없는 부분은 병렬 생성하거나, 작가가 한 장면을 다듬는 동안 다음 장면을 prefetch 생성하는 파이프라이닝을 설계하라. 프롬프트 캐싱의 캐시 히트율 가정(세계관 코어가 캐시된다)을 실측으로 검증하기 전엔 비용 곡선을 확정하지 마라. |
| Med | '정형 필드=결정론 lookup 주입 / 서사 배경=RAG 검색 주입' 원칙(온톨로지·RAG 합의)과 에디터의 '장면 비트 단위 순차 생성'이 결합될 때, 결정론 제약 블록의 주입 시점·범위가 미정이다. 장면마다 전체 제약 블록을 재주입하는가, 회차 단위로 한 번인가. | 온톨로지 합의는 '정형 필드를 constraint 블록으로 프롬프트 상단·캐시에 고정 주입'한다. **사각지대(blind spots)**
- 저작권·프라이버시 처리가 RAG 합의에서 'remainingRisk'로만 언급되고 데이터 거버넌스 설계가 어느 도메인에도 없다. 미공개 원고를 온프레미스 BGE-M3로 임베딩한다지만, 생성 LLM은 Claude/GPT-4.1(외부 API)을 주력으로 쓴다. 즉 임베딩은 온프레미스인데 정작 회차 본문 전체가 생성 프롬프트로 외부 LLM에 전송된다. 작가의 미공개 원고를 외부 API로 보내는 것에 대한 동의·DPA·데이터 보존 정책이 전무하다. 이것이 미해결이면 협력 작가 확보(파일럿 3~5명) 자체가 법무에서 막힌다.
- 삭제·파기 정책이 4개 도메인 어디에도 없다. 작가가 회차를 삭제하거나 작품 전체를 탈퇴하면 RagIndexUnit/ContentChunk(파생 캐시), EntityRevision(append-only 이력), GenerationProvenance(감사 로그), 외부 LLM에 전송된 프롬프트 로그가 어떻게 파기되는가? append-only 이력과 개인정보 파기 의무는 정면으로 충돌한다. 데이터가 코드보다 오래 사는데, 이 제품은 append-only를 여러 곳에서 설계 원칙으로 삼으면서 파기 경로를 한 번도 논하지 않았다.
- 인증·인가·테넌트 격리가 전무하다. 모든 도메인이 'project_id 스코핑으로 격리'를 말하지만 '단일 작가·단일 작품 MVP'를 전제로 owner_id 체크 외 권한 모델이 없다. 파일럿이 3~5명으로 늘면(에디터 합의가 명시) 즉시 멀티테넌트가 되는데, RAG의 workId·온톨로지의 project_id·에디터의 owner_id가 일관된 테넌트 경계를 강제하는지 검증 안 됐다. 한 작가의 회차가 다른 작가의 RAG 검색에 섞이면(IDOR) 미공개 원고 유출이다.
- LLM 비용의 절대 상한·예산 가드레일이 없다. 장면 단위 순차 생성 + 모델 교체 시 수백 회차 전량 재임베딩 + A/B를 위한 3-arm 중복 생성이 겹치면 단일 작가 MVP라도 토큰 비용이 폭증할 수 있다. 작가가 한 회차를 50번 재생성하는 어뷰징(또는 정상적 시행착오)에 대한 rate limit·비용 상한이 어느 도메인에도 없다.
- 롤백·배포 안전성이 논의되지 않았다. 구버전·신버전이 동시에 떠 있을 때(부분 배포) 임베딩 모델 버전이나 청킹 전략이 바뀌면 같은 회차가 두 차원/두 청킹으로 인덱싱돼 검색이 깨진다. IndexState.embeddingModel·schemaVersion 컬럼은 있으나, 마이그레이션 중 신·구 인덱스 호환성 정책이 없다.
- 측정 하니스 자체의 검증이 없다. 원형유지율을 'LCS/문자 유사도'로 측정한다지만, 한국어는 조사·어미 변형이 많아 작가가 '거의 안 고쳤는데' 문자 유사도가 낮게 나오거나 그 반대일 수 있다. 측정 지표가 작가 체감과 상관하는지를 검증하는 메타 단계가 없으면, 측정 그릇이 정밀해도 잘못된 것을 재고 있을 수 있다.
- 관측 가능성(로그/메트릭/트레이스)의 도메인 간 상관관계 추적이 없다. 한 회차 생성이 스토리구조→RAG→에디터→온톨로지를 거치는데, 단일 요청을 4개 도메인에 걸쳐 추적하는 trace_id/correlation_id 전파 규약이 없다. 프로덕션에서 '이 회차 생성이 왜 느렸나/왜 엉뚱한 설정이 들어갔나'를 디버깅하려면 4개 도메인 로그를 수동으로 꿰매야 한다. 각 도메인이 OpenTelemetry/Langfuse를 따로 쓴다고만 적혀 있다.
